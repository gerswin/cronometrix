## Directory Purposes

**`backend/`:**
- Purpose: Rust API server source code
- Contains: HTTP handlers, database layer, authentication, business logic
- Key files: `src/main.rs` (entrypoint), `Cargo.toml` (dependencies), `src/db/migrations/` (schema)

**`backend/src/db/`:**
- Purpose: Database initialization and migrations
- Contains: SQLite local + Turso remote replica setup, migration runner
- Key files: `mod.rs` (init_db, init_db_local, init_db_remote, run_migrations)

**`backend/src/auth/`:**
- Purpose: Authentication and authorization
- Contains: JWT token generation/validation, password hashing, RBAC middleware
- Key files: `service.rs` (token ops), `middleware.rs` (require_auth), `rbac.rs` (role checks)

**`backend/src/employees/`:**
- Purpose: Employee CRUD operations
- Contains: HTTP handlers, service layer, database queries
- Key files: `handlers.rs` (create, list, get, update, deactivate), `service.rs` (business logic)

**`backend/src/departments/`:**
- Purpose: Department management
- Contains: HTTP handlers, service layer for shift + lunch config
- Key files: `handlers.rs`, `service.rs`, `models.rs` (shift_start_time, lunch_mode, base_salary_cents)

**`backend/src/rules/`:**
- Purpose: Global tolerance rules (singleton pattern)
- Contains: GET and PATCH handlers for late_arrival_tolerance, early_departure_tolerance, bonus_minutes
- Key files: `handlers.rs` (get_rules, update_rules), `models.rs`

**`backend/src/setup/`:**
- Purpose: First-time initialization
- Contains: Setup status check, initial admin user creation
- Key files: `handlers.rs` (setup_status, setup_init)

**`frontend/src/app/`:**
- Purpose: Next.js App Router pages
- Contains: Root layout, login page, setup page
- Key files: `layout.tsx` (root), `page.tsx` (home), `login/page.tsx`, `setup/page.tsx`

**`frontend/src/components/`:**
- Purpose: Reusable React components
- Contains: shadcn/ui components (copy-paste), providers wrapper
- Key files: `providers.tsx` (TanStack Query), `ui/button.tsx`, `ui/form.tsx` (React Hook Form)

**`frontend/src/lib/`:**
- Purpose: Shared utilities and API client
- Contains: Axios instance, TanStack Query client, Zod validation schemas
- Key files: `api.ts` (axios + interceptors), `validations.ts` (Zod schemas)

## Naming Conventions

**Files:**

Backend:
- `main.rs` - HTTP server entrypoint
- `lib.rs` - Module root declarations
- `mod.rs` - Module barrel file (public declarations for folder)
- `models.rs` - Request/response DTOs and domain types
- `handlers.rs` - HTTP request/response functions
- `service.rs` - Business logic (queries, validations)
- `middleware.rs` - Tower middleware functions
- `rbac.rs` - Authorization (role checks)
- `*.sql` - Migration scripts in `migrations/` folder

Frontend:
- `page.tsx` - App Router page component
- `layout.tsx` - App Router layout wrapper
- `middleware.ts` - Next.js middleware
- `components/*.tsx` - React components
- `lib/*.ts` - Utility modules
- `*.css` - Global styles

**Directories:**

Backend:
- `src/{domain}/` - Domain module (employees, departments, auth, rules, setup)
- `src/db/` - Database layer
- `tests/` - Integration tests

Frontend:
- `src/app/` - App Router pages (routing via directory structure)
- `src/components/` - Reusable components
- `src/lib/` - Shared utilities
- `public/` - Static assets

**Functions:**

Backend (Rust):
- `snake_case` - Variable and function names (Rust convention)
- `PascalCase` - Struct, enum, trait names
- `SCREAMING_SNAKE_CASE` - Constants
- Handler pattern: `verb_noun()` (e.g., `create_employee`, `list_departments`)
- Service pattern: `verb()` (e.g., `create()`, `list()`, `get_by_id()`)
- Middleware pattern: `require_role()` or `verb_noun()` (e.g., `require_auth`, `require_admin`)

Frontend (TypeScript):
- `camelCase` - Variable and function names (JavaScript convention)
- `PascalCase` - Component and type names
- Handler pattern: `handle<Event>()` (e.g., `handleSubmit`, `handleLogin`)
- Hook pattern: `use<Feature>()` (e.g., `useLoginForm` — not yet used)

**Types:**

Backend:
- Request DTOs: `Create<Entity>Request`, `Update<Entity>Request`, `<Entity>ListQuery`
- Response DTOs: `<Entity>` struct, `Paginated<Entity>` wrapper
- Error types: `AppError` enum

Frontend:
- Schema types: `<FeatureName>Schema` (e.g., `setupSchema`, `loginSchema`)
- Inferred types: `<FeatureName>FormData` (e.g., `SetupFormData`)
- Component props: `<ComponentName>Props` interface (not yet used)

## Where to Add New Code

**New Feature (e.g., Attendance Events):**

Backend:
- Create `backend/src/attendance/` directory
- Add `mod.rs` (module declarations) → `pub mod handlers; pub mod service; pub mod models;`
- Add `models.rs` (AttendanceEvent, CreateAttendanceRequest, etc.)
- Add `service.rs` (business logic: create_attendance, get_by_employee, etc.)
- Add `handlers.rs` (HTTP: `async fn create_attendance(...)`)
- Add routes in `backend/src/main.rs`:
  ```rust
  .route("/attendance", post(attendance::handlers::create_attendance))
  ```
- Add migrations in `backend/src/db/migrations/00X_*.sql` (e.g., 003_attendance_events.sql)

Frontend:
- Create `frontend/src/app/attendance/` directory
- Add `page.tsx` (Attendance list/dashboard page)
- Add `frontend/src/components/AttendanceTable.tsx` (TanStack Table for list)
- Add `frontend/src/lib/useAttendance.ts` hook (TanStack Query for API calls)
- Add Zod schema in `frontend/src/lib/validations.ts` (if form-based)
- Use axios from `lib/api.ts` with Bearer token already attached

**New Component/Module:**

Backend (example: Audit log query):
- Create `backend/src/audit/` directory
- Add service function: `pub async fn get_audit_log(conn: &Connection, record_id: &str) -> Result<Vec<AuditEntry>, AppError>`
- Add handler in existing module that calls the service
- No new routes needed if it's internal usage

Frontend (example: Employee selector dropdown):
- Create `frontend/src/components/EmployeeSelect.tsx`
- Use TanStack Query hook: `const { data: employees } = useQuery({ queryKey: ['employees'], queryFn: () => api.get('/employees') })`
- Export from `components/index.ts` (barrel file)
- Import in parent page/component

**Utilities:**

Backend:
- Helper functions in `backend/src/common.rs` (time formatting, pagination, etc.)
- Middleware in `backend/src/auth/middleware.rs` or new `backend/src/middleware/` folder
- Validators in model structs via `#[validate(...)]` derive macro

Frontend:
- Utility functions in `frontend/src/lib/utils.ts` (classname merging, etc.)
- API helpers in `frontend/src/lib/api.ts` (axios interceptors, query setup)
- Validation in `frontend/src/lib/validations.ts` (Zod schemas)

## Special Directories

**`backend/src/db/migrations/`:**
- Purpose: SQL migration files
- Generated: No (manually written per feature)
- Committed: Yes (version control)
- Naming: `NNN_description.sql` (e.g., 001_initial_schema.sql, 002_audit_triggers.sql)
- Execution: Run once at startup by `run_migrations()` in `backend/src/db/mod.rs`
- Migration tracker: Stored in `_migrations` table to prevent re-running

**`frontend/.next/`:**
- Purpose: Next.js build output and cache
- Generated: Yes (via `next build`)
- Committed: No (.gitignore)
- Cleanup: `rm -rf .next` to force rebuild

**`frontend/node_modules/`:**
- Purpose: Node.js dependencies
- Generated: Yes (via `npm install`)
- Committed: No (.gitignore)
- Lockfile: `package-lock.json` (committed for reproducible installs)

**`.planning/`:**
- Purpose: GSD planning and analysis artifacts
- Generated: Yes (via GSD commands)
- Committed: Yes (version control)
- Contents: Phase roadmaps, codebase analysis docs, research notes

**`.env` files:**
- Purpose: Environment configuration (secrets)
- Generated: No (manually created from `.env.example`)
- Committed: No (.gitignore)
- Backend: `backend/.env` (or `backend/.env.example` as template)
- Frontend: `frontend/.env.local` (not yet in use)

**`public/` (Frontend):**
- Purpose: Static assets served directly
- Generated: No (manually added)
- Committed: Yes
- Contents: favicon.ico, images, logos (if any)
