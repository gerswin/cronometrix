### Step 1: Write the red backend contract

Tests prove:

1. `Claims` deserialization rejects a missing `jti`, and both issuers emit a non-empty UUID `jti`;
2. back-to-back access and refresh tokens have distinct `jti` values;
3. immediate login→refresh produces a different access token and refresh cookie even within one second;
4. replaying the old refresh cookie returns 401 without `Set-Cookie`;
5. two concurrent refresh requests using the same cookie yield exactly one 200 and one 401;
6. the losing 401 does not clear or overwrite the winning cookie.

### Step 2: Add `jti` and one database CAS

Add required `Claims.jti: String`. Both token issuers use `Uuid::new_v4().to_string()`.

Mint and hash the replacement refresh token, then perform exactly one conditional update through the existing write queue:

```sql
UPDATE users
SET refresh_token_hash = ?1, updated_at = unixepoch()
WHERE id = ?2
  AND refresh_token_hash = ?3
  AND status = 'active'
```

Only `rows_affected == 1` may return the access token and replacement cookie. CAS loss/replay returns the existing unauthorized response with no `Set-Cookie` or deletion cookie. Update manual test JWT fixtures to include `jti`; do not make the claim optional.

### Step 2: Implement the single-flight API and provider

Use this public contract:

```ts
export type AuthStatus = 'initializing' | 'authenticated' | 'anonymous'

export interface AuthContextValue {
  status: AuthStatus
  role: 'admin' | 'supervisor' | 'viewer' | null
  sub: string | null
  claims: JWTClaims | null
}

export function refreshAccessToken(): Promise<string>
```

`JWTClaims` includes required `jti` and `token_type`. In `api.ts`, keep exactly one module-level `refreshPromise: Promise<string> | null`. Refresh uses a bare Axios client with credentials and no response interceptor; it resets the promise in `finally`. The response interceptor marks `_retry`, waits for the shared promise, rewrites `Authorization`, and retries once. Request-time expiry handling is deduplicated and becomes eligible again after a later successful token set. Bootstrap failure is silent.

Subscribe to token changes before bootstrapping and guard the asynchronous result with a generation/token snapshot. `SessionGate` renders `data-testid="session-initializing"` until bootstrap settles and redirects from `useEffect` only after anonymous is established.
