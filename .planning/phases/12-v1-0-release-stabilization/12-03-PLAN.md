# Serialized Persistence and Immutable Audit Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: use `superpowers:executing-plans` task-by-task, `superpowers:test-driven-development` for every change, and `superpowers:systematic-debugging` for lock/contention failures.

**Goal:** make `DbWriteQueue` a bounded, observable, drained single-writer boundary; make related domain mutations atomic; eliminate bypass writers; and enforce audit-log immutability in SQLite.

**Architecture:** HTTP requests enqueue for at most five seconds and receive stable 503 errors under saturation. Background workers retry admission only, never an accepted job. Typed queued jobs/transactions return domain values. Filesystem side effects use a compensating guard. Shutdown stops producers, closes admission, drains accepted FIFO work, then awaits the writer. Database triggers reject audit UPDATE/DELETE.

The shutdown contract handles both Ctrl-C and Docker SIGTERM, and an open SSE stream observes cancellation so the writer drain completes within the Compose stop timeout. `AtomicFileGuard` fsyncs the file and parent directory, uses a unique non-clobbering final name, and has duplicate/pre-existing-file tests. Admission errors are separate from typed domain results so 404/409/422 survive queue execution.

**Tech Stack:** Rust/Tokio bounded mpsc, libSQL transactions, Axum errors, SQLite triggers, Python static checker, Bash load profiles.

## Global Constraints

- Depends on 12-01 PASS. It may execute beside 12-02, but 12-02 integrates first; rebase this plan and preserve its final read/payload contracts.
- The existing `.planning/db-write-queue-migration-plan.md` is historical input, not an executable current plan.
- Queue capacity is 1,024 commands. Admission timeout is five seconds.
- Background admission delays are exactly 100 ms, 250 ms, and 500 ms after the initial attempt.
- Never retry a job after the queue accepted it; its commit may have succeeded even if its reply was lost.
- Request saturation maps to HTTP 503 `DB_WRITE_QUEUE_BUSY`; closed/stopped queue maps to 503 `DB_WRITE_QUEUE_UNAVAILABLE`.
- No accepted write may be silently dropped on graceful shutdown.
- Audit rows are append-only. E2E reset must preserve them.
- Do not add coverage exclusions or accept `database is locked` as normal.
- Run narrow tests and affected suites before every commit.

### Step 2: Implement the canonical types

`backend/src/db/write_queue.rs` must expose this behavior and names:

```rust
pub const DEFAULT_WRITE_QUEUE_CAPACITY: usize = 1024;
pub const DEFAULT_ENQUEUE_TIMEOUT: Duration = Duration::from_secs(5);
pub const BACKGROUND_RETRY_DELAYS: [Duration; 3] = [
    Duration::from_millis(100),
    Duration::from_millis(250),
    Duration::from_millis(500),
];

pub struct DbWriteQueueConfig {
    pub capacity: usize,
    pub enqueue_timeout: Duration,
    pub background_retry_delays: [Duration; 3],
}

pub enum DbWriteError {
    Busy,
    Closed,
    WorkerStopped,
    Job(anyhow::Error),
}

pub struct DbWriteQueueStats {
    pub depth: usize,
    pub accepted: u64,
    pub completed: u64,
    pub failed: u64,
    pub busy_rejections: u64,
    pub closed_rejections: u64,
}
```

The queue provides `channel(config)`, `statement(operation, sql, params)`, typed `job(operation, closure)`, typed `transact(operation, closure)`, corresponding `background_statement`, `background_job`, and `background_transact` methods with the same arguments, plus `flush`, `close_and_flush`, and `stats`.

Jobs receive a `QueuedConnection`; transactions receive a `QueuedTransaction`. Their raw connection fields remain private and they expose only required query/statement helpers, so only the queue module constructs a writer.

Use `tokio::sync::mpsc::channel(config.capacity)` and `Sender::reserve` wrapped by the configured timeout. Serialize close/admission with an admission lock so shutdown cannot place its barrier ahead of producers already reserving capacity.

### Step 3: Map stable application errors

Add `AppError::DbWrite(DbWriteError)` mapping:

```text
Busy                 -> 503 DB_WRITE_QUEUE_BUSY
Closed/WorkerStopped -> 503 DB_WRITE_QUEUE_UNAVAILABLE
Job                  -> 500 INTERNAL_ERROR
```

Preserve domain errors returned from closures by converting them before they are erased into queue infrastructure errors; do not turn version/validation/not-found conflicts into 500.

### Step 4: Drain shutdown in producer-first order

The worker processes FIFO until an explicit shutdown command; it must not race a biased `CancellationToken` branch against queued work. In `main.rs`:

1. cancel and await supervisor/recompute/purge/backfill/nightly/renewal producers;
2. call `state.db_write.close_and_flush().await`;
3. await `db_write_handle`;
4. propagate close or join failure;
5. only then exit.

Keep the writer handle; remove `_db_write_handle` fire-and-forget behavior.
