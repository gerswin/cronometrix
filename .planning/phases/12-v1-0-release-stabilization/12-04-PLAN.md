# Private Distribution and Same-Origin Gateway Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: use `superpowers:executing-plans` to execute this plan task-by-task and `superpowers:test-driven-development` before changing runtime behavior.

**Goal:** publish immutable private images to GHCR and produce a checksum-verified installer bundle that exposes Cronometrix through one Nginx origin.

**Architecture:** the browser talks only to the Nginx gateway. Nginx proxies `/api/*` to Axum, including an unbuffered SSE route, and all other paths to Next.js. Cloudflare Tunnel points only to `gateway:8080`. GitHub Actions builds three private images, captures their digests, and packages `install.sh`, Compose, the release manifest, and checksums as a private artifact. The installer authenticates to GHCR with a per-installation read-only credential and deploys digest-pinned images with rollback.

**Tech Stack:** Next.js 16, Axios/EventSource, Nginx, Docker Buildx/Compose, GitHub Actions, GHCR, Bash.

## Global Constraints

- Start only after 12-02 and 12-03 are merged with their scoped checkpoints
  passing. Their project-wide coverage ledger remains open until 12-05.
- Production browser requests must not contain `http://api:3001` or `http://localhost:3001`.
- Do not publish the installer or Compose manifest at an anonymous URL.
- Never print or commit GHCR tokens, Cloudflare tokens, license keys, admin passwords, JWT secrets, device keys, or Docker auth JSON.
- Production image references must include `@sha256:`. Mutable tags are metadata only and are never the deployed identity.
- v1.0 release images target `linux/amd64`; the workflow builds that platform explicitly and the installer rejects other OS/architectures.
- The bundle source SHA, all image digests, and the bundle checksum must be mutually traceable.
- The installer must preserve an existing data directory and secrets, validate configuration before replacement, and roll back the prior manifest when health checks fail.
- Run tests before each commit.

export function resolvePublicApiBase(raw: string | undefined): string {
  return raw === undefined ? 'http://localhost:3001' : raw
}

export function resolveInternalApiBase(
  internal: string | undefined,
  publicBase: string | undefined,
): string {
  if (internal) return internal
  const resolvedPublic = resolvePublicApiBase(publicBase)
  return resolvedPublic || 'http://localhost:3001'
}

### Step 3: Create the release workflow

Create `.github/workflows/release.yml` with these binding properties:

- workflow name: `Release`;
- triggers: pushes to branches matching `codex/release-build-*` and tags matching `v*`; candidate builds do not use `workflow_dispatch`, which cannot bootstrap a workflow that is absent from the default branch;
- default permissions: `contents: read`; grant `packages: write` only to `build-images` and `promote-images`, grant `checks: read` only where promotion verifies checks, and require the promotion job's protected environment approval;
- concurrency group: `release-${{ github.ref }}` with `cancel-in-progress: false`;
- one Buildx matrix job for `api`, `web`, and `gateway`, executed only for a branch push. After checkout it resolves `SOURCE_SHA="$(git rev-parse HEAD^{commit})"` and fails unless both `GITHUB_SHA == SOURCE_SHA` and `GITHUB_REF_NAME == codex/release-build-${SOURCE_SHA}`. It builds `linux/amd64`, sets `RELEASE_VERSION=sha-${SOURCE_SHA:0:12}`, and records that full SHA as the OCI revision;
- one promotion matrix job for tag events; it resolves the existing `sha-${SOURCE_SHA:0:12}` manifests where `SOURCE_SHA=$(git rev-parse HEAD^{commit})`, verifies their OCI revision label equals the tagged commit and that all five required checks succeeded on that SHA, then adds aliases without rebuilding layers;
- image names `ghcr.io/gerswin/cronometrix-api`, `ghcr.io/gerswin/cronometrix-web`, and `ghcr.io/gerswin/cronometrix-gateway`;
- build-mode tags `sha-${SOURCE_SHA:0:12}`; promotion mode adds the Git tag to those exact existing digests;
- the web build argument `NEXT_PUBLIC_API_URL=`;
- upload one small digest artifact per matrix image;
- a bundle job that consumes digest artifacts from either build or promotion, uses the fixed Cloudflare digest above, writes `release-manifest.env`, runs `scripts/verify-release-manifest.sh`, writes an internal `SHA256SUMS` covering the other four bundle files, creates `cronometrix-${RELEASE_VERSION}-${SOURCE_SHA:0:12}.tar.gz` containing all five files, writes an external `.tar.gz.sha256`, verifies both checksum layers, and uploads the tarball plus its external checksum as a private Actions artifact with 14-day retention;
- use `docker/login-action@v3`, `docker/setup-buildx-action@v3`, `docker/build-push-action@v6`, and `actions/upload-artifact@v4`;
- do not echo `${{ secrets.GITHUB_TOKEN }}` or generated Docker auth.

The trigger, event partition, and least-privilege skeleton are exact. Quote
`'on'` so PyYAML-based static validation does not coerce the key to boolean:

```yaml
'on':
  push:
    branches:
      - 'codex/release-build-*'
    tags:
      - 'v*'

permissions:
  contents: read

jobs:
  build-images:
    if: github.event_name == 'push' && startsWith(github.ref, 'refs/heads/codex/release-build-')
    permissions:
      contents: read
      packages: write
  promote-images:
    if: github.event_name == 'push' && startsWith(github.ref, 'refs/tags/v')
    environment: release
    permissions:
      contents: read
      checks: read
      packages: write
  bundle:
    needs: [build-images, promote-images]
    if: >-
      always() &&
      ((needs.build-images.result == 'success' && needs.promote-images.result == 'skipped') ||
       (needs.build-images.result == 'skipped' && needs.promote-images.result == 'success'))
    permissions:
      contents: read
```

The `release` environment must exist with required reviewers before any tag
promotion. Branch candidate builds never enter that environment. The bundle
must see exactly one successful upstream and one skipped upstream; failure,
cancellation, two successes, or two skips all block it.

The bundle must contain exactly:

```text
install.sh
docker-compose.yml
release-manifest.env
nginx.conf
SHA256SUMS
```

`SOURCE_SHA` must equal the resolved commit SHA (`git rev-parse HEAD^{commit}`), not an annotated tag object. In build mode each application image uses the digest emitted by its own `linux/amd64` build step and carries the same OCI revision. In promotion mode each digest must equal the pre-existing SHA-tag digest and the job must fail if revision metadata or the five-check rollup differs. This plan proves the publisher with a dry-release SHA. After 12-05 Tasks 1–4 add their own commits, 12-05 must publish its newer final candidate SHA through this same workflow and wait for the full CI rollup on that newer SHA before Phase 13 consumes its bundle. A final tag may never rebuild an image that live validation tested under another digest.

### Step 3: Implement transactional deployment and rollback

Use the verified release manifest as the sole source of image references. Before replacing an existing release:

1. create a timestamped backup directory under `${INSTALL_DIR}/releases/rollback/`;
2. take a consistent SQLite/WAL backup and copy the current Compose, Nginx, and release manifest there;
3. record the current container image IDs with `docker compose images --format json`;
4. install the candidate files atomically using temporary files plus `mv`;
5. run `docker compose --env-file .env --env-file release-manifest.env config --quiet`;
6. pull all four digest-pinned images;
7. start `api`, wait for its internal health, then start `web` and `gateway`; keep `cloudflared` stopped until local probes pass;
8. probe gateway health, API health through gateway, setup status through gateway, upload limits, and container health; only then start `cloudflared`;
9. if any step fails, restore the backed-up files, start the previous release, verify its health, and exit non-zero;

### Step 2: Trigger a private dry release from the branch SHA

Push one immutable, one-use release-build ref and wait for the push-triggered run. This works before `release.yml` exists on the default branch. Download the artifact with authenticated `gh`, without extracting it inside the repository:

```bash
set -euo pipefail
test "$(git status --porcelain)" = ""
SOURCE_SHA="$(git rev-parse HEAD^{commit})"
test -n "$SOURCE_SHA"
RELEASE_BUILD_BRANCH="codex/release-build-${SOURCE_SHA}"
test -z "$(git ls-remote --heads origin "refs/heads/${RELEASE_BUILD_BRANCH}")"
TRIGGERED_AT="$(date -u +%Y-%m-%dT%H:%M:%SZ)"
git push origin "${SOURCE_SHA}:refs/heads/${RELEASE_BUILD_BRANCH}"
test "$(git ls-remote origin "refs/heads/${RELEASE_BUILD_BRANCH}" | awk '{print $1}')" = "$SOURCE_SHA"
RUN_ID=""
for attempt in $(seq 1 30); do
  RUN_ID="$(gh run list --branch "$RELEASE_BUILD_BRANCH" --event push --limit 50 \
    --json databaseId,headSha,createdAt,workflowName \
    --jq "[.[] | select(.workflowName == \"Release\" and .headSha == \"$SOURCE_SHA\" and .createdAt >= \"$TRIGGERED_AT\")] | .[0].databaseId // empty")"
  test -n "$RUN_ID" && break
  sleep 5
done
test -n "$RUN_ID"
gh run watch "$RUN_ID" --exit-status
TMP_RELEASE="$(mktemp -d)"
gh run download "$RUN_ID" --dir "$TMP_RELEASE"
test "$(find "$TMP_RELEASE" -name '*.tar.gz' | wc -l)" = 1
test "$(find "$TMP_RELEASE" -name '*.tar.gz.sha256' | wc -l)" = 1
find "$TMP_RELEASE" -name '*.tar.gz.sha256' -execdir sha256sum -c "{}" \;
EXTRACTED_RELEASE="$(mktemp -d)"
RELEASE_TAR="$(find "$TMP_RELEASE" -name '*.tar.gz' -print -quit)"
tar -tzf "$RELEASE_TAR" | sort | diff -u <(printf '%s\n' SHA256SUMS docker-compose.yml install.sh nginx.conf release-manifest.env) -
tar -xzf "$RELEASE_TAR" -C "$EXTRACTED_RELEASE"
(cd "$EXTRACTED_RELEASE" && sha256sum -c SHA256SUMS)
rg -n "^SOURCE_SHA=${SOURCE_SHA}$" "$EXTRACTED_RELEASE/release-manifest.env"
rm -rf "$EXTRACTED_RELEASE"
rm -rf "$TMP_RELEASE"
```

Expected: workflow succeeds, checksums pass, and the manifest source SHA equals the branch SHA. Do not commit downloaded images, bundles, auth files, or source-bearing reports.
