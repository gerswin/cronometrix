# Live CI Gates and Branch Protection Validation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: use `superpowers:executing-plans`; all GitHub mutations in this plan are deliberate validation actions and must be linked in the summary.

**Goal:** prove that the exact Phase 12 candidate passes every required check, that deliberate regressions are rejected, and that `main` cannot merge without the required checks.

**Architecture:** validate an immutable candidate PR first, configure and re-read branch protection before creating the red-regression PR, then prove that the protected branch blocks that PR, close and delete it without merge. Evidence is URLs plus redacted JSON/text; no source-bearing CI reports are committed.

**Tech Stack:** GitHub Actions, GitHub CLI/API, git, jq.

## Global Constraints

- Begin only after `12-05-SUMMARY.md` says `Verdict: PASS`.
- Set `CANDIDATE_SHA` once from remote branch `codex/v1-release-candidate` and never move that branch during this plan.
- Commit all validation evidence on a separate `codex/v1-live-validation-evidence-*` branch.
- If a source commit changes, stop and restart the candidate validation.
- If `origin/main` moves after `BASE_SHA` is captured, stop; rebase/remediate and repeat Phase 12/13 evidence on a new candidate rather than merging an untested composition.
- Never merge the deliberate red PR. Close it and delete its remote branch after evidence capture.
- Do not disable, soften, or temporarily bypass any CI threshold.
- Preserve private Actions reports only in GitHub for 14 days; commit URLs, conclusions, names, SHAs, and hashes only.
- Branch protection is an external mutation. Capture the prior configuration before changing it and do not remove stricter existing protections.

## Task 1: Prove all candidate checks green

**Files:**

- Create: `.planning/phases/13-v1-0-live-validation/evidence/01-ci/candidate.json`
- Create: `.planning/phases/13-v1-0-live-validation/evidence/01-ci/checks-green.json`
- Create: `.planning/phases/13-v1-0-live-validation/evidence/01-ci/README.md`

### Step 1: Verify the local candidate identity

Fetch the immutable candidate and create a separate evidence worktree:

```bash
git fetch origin main codex/v1-release-candidate codex/v1-release-evidence-base
CANDIDATE_BRANCH=codex/v1-release-candidate
CANDIDATE_SHA="$(git rev-parse "origin/${CANDIDATE_BRANCH}")"
BASE_SHA="$(git rev-parse origin/main)"
SUMMARY_SHA="$(git rev-parse origin/codex/v1-release-evidence-base)"
git merge-base --is-ancestor "$CANDIDATE_SHA" "$SUMMARY_SHA"
git merge-base --is-ancestor "$BASE_SHA" "$CANDIDATE_SHA"
SHORT_SHA="$(printf '%s' "$CANDIDATE_SHA" | cut -c1-12)"
EVIDENCE_BRANCH="codex/v1-live-validation-evidence-${SHORT_SHA}"
EVIDENCE_TREE="$(mktemp -d)/cronometrix-v1-evidence"
git worktree add -b "$EVIDENCE_BRANCH" "$EVIDENCE_TREE" "$SUMMARY_SHA"
printf 'candidate=%s\nbase=%s\nsummary=%s\n' "$CANDIDATE_SHA" "$BASE_SHA" "$SUMMARY_SHA"
```

Expected: clean worktree, candidate descends from current `origin/main`, and both full SHAs are printed.

### Step 2: Push the candidate and open or reuse its PR

```bash
git push origin "$CANDIDATE_BRANCH"
PR_URL="$(gh pr view "$CANDIDATE_BRANCH" --json url --jq .url 2>/dev/null || \
  gh pr create --base main --head "$CANDIDATE_BRANCH" \
    --title "v1.0 release stabilization" \
    --body "Executes the approved Phase 12 remediation plan. Live validation evidence is tracked under .planning/phases/13-v1-0-live-validation/." \
    --draft)"
printf '%s\n' "$PR_URL"
```

Expected: one draft PR targeting `main`, with head SHA equal to `CANDIDATE_SHA`.

Before creating any red-regression branch, execute Task 3 Steps 1–3 below: capture, normalize, apply, and re-read branch protection. The effective protection must require at least one approving review (preserving any stricter existing count), the five named checks, conversation resolution, and no force pushes/deletions. If existing `required_linear_history` is true, stop because the approved merge-commit promotion cannot preserve the tested candidate ancestry.

### Step 3: Wait for the required checks

```bash
gh pr checks "$PR_URL" --watch --fail-fast
gh pr view "$PR_URL" --json number,url,baseRefName,headRefName,headRefOid,isDraft,statusCheckRollup \
  > /tmp/cronometrix-candidate.json
jq -e --arg sha "$CANDIDATE_SHA" '
  .baseRefName == "main" and
  .headRefOid == $sha and
  ([.statusCheckRollup[] | select(.name == "Backend Coverage" or .name == "Frontend Coverage" or .name == "E2E Tests" or .name == "Container Smoke" or .name == "Release Gate")] | map(select(.conclusion == "SUCCESS")) | length == 5)
' /tmp/cronometrix-candidate.json
```

The required successful check names are:

```text
Backend Coverage
Frontend Coverage
E2E Tests
Container Smoke
Release Gate
```

If Plan 12-05 deliberately consolidated jobs, update this list in both this plan and branch protection in the same planning-only commit before execution; never infer names while applying protection.

Expected: every named check exists exactly once and concludes `SUCCESS` on `CANDIDATE_SHA`.

### Step 4: Commit redacted evidence

In `EVIDENCE_TREE`, copy only the safe fields from `/tmp/cronometrix-candidate.json` into `candidate.json` and add the captured `base_sha`. Write `checks-green.json` as an array of check name, conclusion, details URL, workflow run ID, started time, completed time, and candidate SHA. Write `README.md` with `Verdict: PASS`, PR URL, base/candidate SHAs, and capture time.

Validate:

```bash
cd "$EVIDENCE_TREE"
jq -e --arg sha "$CANDIDATE_SHA" '.headRefOid == $sha' \
  .planning/phases/13-v1-0-live-validation/evidence/01-ci/candidate.json
jq -e 'length == 5 and all(.conclusion == "SUCCESS")' \
  .planning/phases/13-v1-0-live-validation/evidence/01-ci/checks-green.json
rg -n 'Verdict: PASS' \
  .planning/phases/13-v1-0-live-validation/evidence/01-ci/README.md
```

Expected: all commands exit 0.

Commit:

```bash
git add .planning/phases/13-v1-0-live-validation/evidence/01-ci
git commit -m "docs(validation): record green candidate checks"
```

The evidence commit exists only on `EVIDENCE_BRANCH`; the candidate branch and SHA remain unchanged.

### Step 1: Create an isolated throwaway worktree from the tested candidate

RED_BRANCH="codex/validate-gates-red-${SHORT_SHA}"

Expected: clean worktree at exactly `CANDIDATE_SHA`.

### Step 2: Add three explicit regressions

Create `backend/src/coverage_regression_probe.rs`:

```rust
pub fn uncovered_branch(value: i32) -> i32 {
    if value > 10 {
        value * 2
    } else if value < 0 {
        value.abs()
    } else {
        value + 1
    }
}
```

Export it from `backend/src/lib.rs` with `pub mod coverage_regression_probe;` and add no test.

Create `frontend/src/lib/coverage-regression-probe.ts`:

```ts
export function uncoveredBranch(value: number): number {
  if (value > 10) return value * 2
  if (value < 0) return Math.abs(value)
  return value + 1
}
```

Add no test.

Create `frontend/e2e/gate-regression.spec.ts`:

```ts
import { expect, test } from '@playwright/test'

test('deliberate validation regression', async () => {
  expect('red-gate-probe').toBe('this-must-fail')
})
```

Commit and push:

```bash
git add backend/src/coverage_regression_probe.rs backend/src/lib.rs \
  frontend/src/lib/coverage-regression-probe.ts frontend/e2e/gate-regression.spec.ts
git commit -m "test(validation): deliberately break release gates"
git push -u origin "$RED_BRANCH"
RED_PR_URL="$(gh pr create --base main --head "$RED_BRANCH" \
  --title "DO NOT MERGE: deliberate CI gate regression" \
  --body "Validation-only PR. It must fail backend coverage, frontend coverage, and E2E. It will be closed and deleted without merge.")"
```

### Step 3: Wait for and assert the failures

```bash
gh pr checks "$RED_PR_URL" --watch || true
gh pr view "$RED_PR_URL" --json number,url,headRefOid,statusCheckRollup > /tmp/cronometrix-red.json
jq -e '
  ([.statusCheckRollup[] | select(.name == "Backend Coverage") | .conclusion] | any(. == "FAILURE")) and
  ([.statusCheckRollup[] | select(.name == "Frontend Coverage") | .conclusion] | any(. == "FAILURE")) and
  ([.statusCheckRollup[] | select(.name == "E2E Tests") | .conclusion] | any(. == "FAILURE"))
' /tmp/cronometrix-red.json
```

Expected: the three targeted jobs conclude `FAILURE`. Additional failing jobs are acceptable only when directly caused by these explicit probe files; document them.

Open each workflow log and record the safe failure marker:

- backend: per-file line or branch coverage below its floor for `coverage_regression_probe.rs`;
- frontend: per-file threshold below its floor for `coverage-regression-probe.ts`;
- E2E: expected `red-gate-probe` to equal `this-must-fail`.

### Step 4: Prove protection blocks the red PR, then close and delete it

```bash
gh pr view "$RED_PR_URL" --json mergeStateStatus,reviewDecision > /tmp/cronometrix-red-protected.json
jq -e '.mergeStateStatus == "BLOCKED" and (.reviewDecision != "APPROVED")' /tmp/cronometrix-red-protected.json
gh pr close "$RED_PR_URL" --comment "Validation complete: required gates failed and branch protection blocked merge. Closing without merge." --delete-branch
gh pr view "$RED_PR_URL" --json number,url,state,mergedAt,headRefOid,statusCheckRollup \
  > /tmp/cronometrix-red-closed.json
cd -
git worktree remove "$RED_TREE"
git branch -D "$RED_BRANCH"
```

Expected: PR state `CLOSED`, `mergedAt` null, and remote branch absent.

### Step 5: Commit evidence on the validation branch

Return to `EVIDENCE_TREE`. Write PR state from `/tmp/cronometrix-red-closed.json` and check details from `/tmp/cronometrix-red.json`, plus failure markers and run URLs, under `evidence/02-red-regression/`.
