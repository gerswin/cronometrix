# v1.0 Evidence Reconciliation and Release Sign-off Plan

> **For agentic workers:** REQUIRED SUB-SKILL: use `superpowers:executing-plans`; the release/tag checkpoint requires explicit operator approval and must not be auto-resolved.

**Goal:** reconcile all Phase 12 and 13 evidence against one candidate, preserve the two accepted deferrals, merge the protected candidate, and promote the already tested image digests as v1.0.0.

**Architecture:** a machine-readable verifier checks SHA and digest continuity across plan summaries, GitHub checks, the release manifest, and fresh-VM evidence. The evidence branch updates planning truth separately from product code. After explicit approval, the protected candidate PR merges with its head commit preserved as a parent, `v1.0.0` tags that exact tested commit, and the tag workflow promotes—not rebuilds—the tested image digests.

**Tech Stack:** Python 3, git, GitHub CLI/API, GitHub Actions, GHCR.

## Global Constraints

- Depends on `13-01-SUMMARY.md` and `13-02-SUMMARY.md` both recording `Verdict: PASS`.
- The candidate product SHA and four tested image digests must not change.
- Keep the validation evidence branch separate from the candidate branch until after v1.0.0 promotion.
- Never turn `HIKVISION-LIVE` or `LIC-05-CROSS-HOST` into PASS. They remain accepted, owned, time-bounded deferrals.
- Do not lower branch protection, use an admin bypass, merge a red PR, or retag an existing release tag.
- The tag workflow must promote the digest-pinned images tested in Plan 13-02. Rebuilding is a different artifact and fails traceability.
- Creating or pushing `v1.0.0` requires explicit operator approval at Task 4.

### Step 2: Implement strict consistency checks

- require `Verdict: PASS` in every summary;

- require both deferred IDs to exist with status `DEFERRED`, owner, due condition, and review date;
- reject duplicate fields, missing files, ambiguous multiple SHAs, invalid digests, and any `PASS` assigned to a deferred ID;
- print paths and field names on failure, never full evidence contents.

## Task 4: Blocking approval, protected merge, and immutable promotion

**Files:** no source files before approval; GitHub state changes only.

### Step 1: Request explicit approval

Present the release decision packet, candidate SHA, image digests, fresh-VM verdict, and two deferrals to the operator. Ask exactly whether to release `v1.0.0` with those two deferred risks.

Do not merge, tag, push, dispatch, or promote until the operator explicitly answers yes. Silence, prior planning approval, or approval of Phase 12 is not release approval.

### Step 2: Recheck the immutable candidate after approval

```bash
CANDIDATE_SHA="$(jq -r .candidate_sha \
  .planning/phases/13-v1-0-live-validation/evidence/06-signoff/release-candidate.json)"
BASE_SHA="$(jq -r .base_sha \
  .planning/phases/13-v1-0-live-validation/evidence/06-signoff/release-candidate.json)"
CANDIDATE_PR_URL="$(jq -r .candidate_pr_url \
  .planning/phases/13-v1-0-live-validation/evidence/06-signoff/release-candidate.json)"
git fetch origin main
test "$(git rev-parse origin/main)" = "$BASE_SHA"
gh pr view "$CANDIDATE_PR_URL" --json headRefOid,mergeable,mergeStateStatus,isDraft,reviewDecision,statusCheckRollup \
  > /tmp/cronometrix-final-pr.json
jq -e --arg sha "$CANDIDATE_SHA" '
  .headRefOid == $sha and
  .mergeable == "MERGEABLE" and
  .mergeStateStatus == "CLEAN" and
  .reviewDecision == "APPROVED" and
  ([.statusCheckRollup[].conclusion] | all(. == "SUCCESS"))
' /tmp/cronometrix-final-pr.json
```

Expected: exact SHA, mergeable, clean, all checks success.

### Step 3: Merge without rewriting the tested head commit

Mark the PR ready if still draft, then merge with a merge commit:

```bash
gh pr ready "$CANDIDATE_PR_URL" 2>/dev/null || true
gh pr merge "$CANDIDATE_PR_URL" --merge --delete-branch
git fetch origin main --tags
git merge-base --is-ancestor "$CANDIDATE_SHA" origin/main
```

The branch protection created by 13-01 must allow merge commits so the tested candidate remains an ancestor with the same SHA. If repository policy forces squash/rebase, stop and redesign promotion; do not release an untested rewritten SHA.

### Step 4: Create the immutable tag on the tested candidate

First ensure the tag does not exist locally or remotely:

```bash
! git show-ref --verify --quiet refs/tags/v1.0.0
! git ls-remote --exit-code --tags origin refs/tags/v1.0.0
git tag -a v1.0.0 "$CANDIDATE_SHA" \
  -m "Cronometrix v1.0.0 — validated release candidate ${CANDIDATE_SHA}"
git push origin refs/tags/v1.0.0
```

Expected: a new annotated tag points exactly to `CANDIDATE_SHA`.

### Step 5: Verify promotion uses tested digests

The tag triggers `.github/workflows/release.yml` in promotion mode. It must resolve the existing `sha-${CANDIDATE_SHA::12}` image tags, verify their OCI revision label equals `CANDIDATE_SHA`, confirm all five required checks succeeded on that SHA, add the `v1.0.0` aliases to those manifests, and build a final private installer bundle without rebuilding image layers.

```bash
TAG_RUN_ID="$(gh run list --workflow release.yml --branch v1.0.0 --limit 1 --json databaseId --jq '.[0].databaseId')"
gh run watch "$TAG_RUN_ID" --exit-status
TAG_TMP="$(mktemp -d)"
gh run download "$TAG_RUN_ID" --dir "$TAG_TMP"
find "$TAG_TMP" -name '*.tar.gz.sha256' -execdir sha256sum -c "{}" \;
TAG_EXTRACTED="$(mktemp -d)"
tar -xzf "$(find "$TAG_TMP" -name '*.tar.gz' -print -quit)" -C "$TAG_EXTRACTED"
(cd "$TAG_EXTRACTED" && sha256sum -c SHA256SUMS)
python3 scripts/verify-v1-release-evidence.py \
  --phase12 .planning/phases/12-v1-0-release-stabilization \
  --phase13 .planning/phases/13-v1-0-live-validation \
  > /tmp/tested-release.json
for name in api web gateway cloudflared; do
  TESTED="$(jq -r --arg name "$name" '.image_digests[$name]' /tmp/tested-release.json)"
  rg -n "${TESTED}" "$TAG_EXTRACTED/release-manifest.env"
done
rm -rf "$TAG_EXTRACTED"
rm -rf "$TAG_TMP"
```

Expected: tag workflow succeeds and all four final manifest digests equal the tested digests.

If any digest differs, mark the release workflow failed, delete no tag without explicit incident direction, and do not call v1.0.0 released.
