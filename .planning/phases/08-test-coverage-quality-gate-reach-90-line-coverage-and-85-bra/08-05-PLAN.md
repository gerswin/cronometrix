    **The workflow content** — start from the verbatim skeleton in 08-RESEARCH.md lines 491-570 and adjust per the items below:

    ```yaml
    # .github/workflows/ci.yml
    # Phase 8 quality gate: backend + frontend coverage on every push + PR to main.
    # Hard-fails on threshold miss (D-13). HTML artifacts uploaded for triage (D-04).
    name: CI

    on:
      push:
        branches: ['**']
      pull_request:
        branches: [main]

    permissions:
      contents: read

    jobs:
      backend-coverage:
        name: Backend Coverage
        runs-on: ubuntu-latest
        env:
          CARGO_TERM_COLOR: always
        defaults:
          run:
            working-directory: backend
        steps:
          - uses: actions/checkout@v4
          - name: Install nightly Rust (--branch is unstable)
            run: |
              rustup toolchain install nightly --component llvm-tools-preview
              rustup default nightly
              rustc --version
          - uses: Swatinem/rust-cache@v2
            with:
              workspaces: backend
          - name: Install cargo-llvm-cov
            uses: taiki-e/install-action@v2
            with:
              tool: cargo-llvm-cov@0.8.5
          - name: Install cargo-nextest
            uses: taiki-e/install-action@v2
            with:
              tool: cargo-nextest
          - name: Run coverage with project-wide line gate
            run: |
              cargo llvm-cov nextest \
                --branch --all-features \
                --ignore-filename-regex '(main\.rs|tests/common/.*)' \
                --fail-under-lines 90 \
                --lcov --output-path lcov.info
          - name: Enforce branch + per-file thresholds
            run: bash ../scripts/enforce-coverage-floor.sh lcov.info 85 70 60
          - name: Generate HTML report
            if: always()
            run: cargo llvm-cov --branch --all-features --no-clean --html
          - name: Upload HTML artifact
            if: always()
            uses: actions/upload-artifact@v4
            with:
              name: backend-coverage-html
              path: backend/target/llvm-cov/html
              retention-days: 14

      frontend-coverage:
        name: Frontend Coverage
        runs-on: ubuntu-latest
        defaults:
          run:
            working-directory: frontend
        steps:
          - uses: actions/checkout@v4
          - uses: actions/setup-node@v4
            with:
              node-version: 20
              cache: npm
              cache-dependency-path: frontend/package-lock.json
          - run: npm ci
          - name: Run Vitest with coverage (thresholds enforced via vitest.config.ts)
            run: npx vitest run --coverage
          - name: Upload HTML artifact
            if: always()
            uses: actions/upload-artifact@v4
            with:
              name: frontend-coverage-html
              path: frontend/coverage
              retention-days: 14
    ```

    - `actions/upload-artifact@v4`: REQUIRED — v3 deprecated.
    - `actions/setup-node@v4`: REQUIRED.

    - `Swatinem/rust-cache@v2`: standard.
    - `cargo-llvm-cov@0.8.5`: pinned to match local development version (rust-toolchain.toml drift on cargo-llvm-cov is undesirable).

    **Exclusion regex parity:** If Plan 04 added any new patterns to the Makefile's `--ignore-filename-regex` (e.g., `'(main\.rs|tests/common/.*|<plan-04-additions>)'`), the same exact regex MUST be used in the CI's `Run coverage` step.     **Working directory note:** `defaults.run.working-directory: backend` makes every step's `cd backend &&` implicit. The post-processor invocation must use `bash ../scripts/enforce-coverage-floor.sh lcov.info 85 70 60` (with `../` because we're running from `backend/`).

    **`if: always()` on artifact upload:** intentional — uploads even when the threshold step failed, so the developer can drill into the failing report. RESEARCH § CI Workflow Skeleton confirms this idiom.

    **Branch protection setup (NOT IN THIS PLAN'S FILES):** Both jobs must be configured as REQUIRED status checks via GitHub UI Branch Protection rules for `main`. <threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub → workflow runner | Trusted CI runtime (ubuntu-latest); ephemeral; no persistent secrets needed for this workflow |
| workflow → external action | Pinned actions (`@v4`, `@v2`) constrain version drift |
| workflow → GITHUB_TOKEN | Scoped to `contents: read` (least privilege); cannot mutate repo or PRs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-14 | Tampering | Pinned-action drift / supply-chain | mitigate | All `uses:` pin to a specific major tag (`@v4`, `@v2`, `cargo-llvm-cov@0.8.5`). Per RESEARCH § Security Domain, executor MAY upgrade pins to commit SHAs if the project's supply-chain posture later requires it; major-tag pin is the standard baseline. |
| T-08-15 | Elevation of Privilege | GITHUB_TOKEN scope | mitigate | `permissions: contents: read` set at workflow level. The workflow does not need write access to PR comments, contents, or anything else — it only reads source and uploads artifacts (artifacts use the action's own implicit token, not the workflow's). |
| T-08-16 | Information Disclosure | HTML artifact contains source | accept | Repo is private; artifacts scoped to repo collaborators with read access. CLAUDE.md (Plan 06) will document this so a future open-sourcing decision can revisit retention/visibility. |
| T-08-17 | Tampering | Untrusted PR runs CI with secrets | mitigate | This workflow uses NO repository secrets — there is nothing to leak via a malicious PR. `pull_request` (not `pull_request_target`) is used, so PRs from forks run in the fork's restricted context. |
| T-08-18 | Tampering | Nightly Rust pulled at runtime | mitigate | `rustup toolchain install nightly` is run AFTER `rust-toolchain.toml` is checked out, so the pinned date in `rust-toolchain.toml` (Plan 03) determines the actual nightly used. Bare `nightly` is never installed without the pin. |
</threat_model>
