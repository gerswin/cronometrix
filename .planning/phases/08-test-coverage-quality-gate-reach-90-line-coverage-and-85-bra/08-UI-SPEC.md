# Phase 8 — UI Design Contract

> NO-OP CONTRACT — Phase 8 introduces no user-facing UI. This document exists to satisfy the
> safety gate triggered by the word "frontend" in the phase title (Vitest+v8 coverage tooling)
> and to forbid planner/executor from inventing UI variations under the guise of "test fixtures".
>
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope of This Phase

**No new UI surfaces.** This phase delivers:

1. Coverage tooling configuration (`frontend/vitest.config.ts` extension with `coverage.thresholds`,
   `coverage.thresholds.perFile`, `coverage.include`, `coverage.exclude`).
2. Gap-fill unit/integration tests under `frontend/src/__tests__/` and co-located
   `*.test.tsx` files for **already-shipped** components, hooks, and lib modules in
   `frontend/src/components/`, `frontend/src/hooks/`, `frontend/src/lib/`.
3. Backend `cargo-llvm-cov` instrumentation, `Makefile`, `scripts/enforce-coverage-floor.sh`,
   `.github/workflows/ci.yml`, and the `AppState`/`Paths` filesystem-root injection refactor
   (zero UI impact — backend-only).
4. Documentation appended to root `CLAUDE.md`.

The phase **does not** add, change, or design any user-facing screens, components,
layouts, forms, modals, banners, dialogs, dropdowns, table presentations, copy strings,
icons, color tokens, spacing tokens, or visual elements of any kind.

---

## Locked Rule (binding on planner + executor)

Gap-fill tests **MUST**:

- Import the existing exported component/hook/util **as-is** from its current module path.
- Mount components without modifying their props beyond the props the component already accepts.
- Assert against existing rendered output (text, role, label, structure) — never alter copy,
  layout, classNames, or styling to make a test easier to write.
- Use the existing design tokens (CSS variables in `frontend/src/app/globals.css`) and the
  existing `tailwind` + `shadcn/ui` stack as inherited.
- Use existing icon imports from `lucide-react` and existing test infra (`@testing-library/react`,
  `@testing-library/jest-dom`, `msw`).

Gap-fill tests **MUST NOT**:

- Add new styled wrappers around components solely "for testability".
- Introduce new copy strings, button labels, empty-state messages, error messages, headings,
  or accessible names that do not already exist in the production code.
- Add new layout variants, density modes, theme variants, or color schemes.
- Add new design tokens (CSS variables, Tailwind theme extensions, font sizes, spacing values).
- Add or modify shadcn registry entries.
- Change `frontend/src/app/globals.css`, `frontend/components.json`, or any tailwind config.
- Modify components in `frontend/src/components/ui/` (vendored shadcn copies — excluded from
  coverage per CONTEXT D-10 anyway).

## Allowed Test-Only Artifacts

These are **not UI** and do not violate the no-op contract:

| Artifact | Path Pattern | Purpose |
|----------|--------------|---------|
| Test files | `frontend/src/**/*.test.{ts,tsx}` | Co-located component / hook / util tests |
| Test files | `frontend/src/**/*.spec.{ts,tsx}` | Same, alternate naming |
| Test fixtures | `frontend/src/__tests__/fixtures/**` | Plain TS/JSON data shapes used by multiple tests |
| MSW handlers | `frontend/src/__tests__/msw/**` | Mock API response handlers (test infra only) |
| Test setup | `frontend/src/__tests__/setup.ts` | Already exists — extend if needed, don't recreate |
| Vitest config | `frontend/vitest.config.ts` | Extended with `coverage` block |

These files **are excluded from the coverage denominator** by the `coverage.exclude` glob set
defined in 08-RESEARCH.md, so they do not artificially inflate or deflate the coverage number.

## Visual States (Inherited — No Changes This Phase)

Tests verify the **already-shipped** states of existing components by mounting them with
appropriate prop combinations:

|-------|-------------------|-----------|

| Disabled | Render with `disabled={true}` | Adding new disabled styling |

## Accessibility (Inherited — No Changes This Phase)

- Tests **may** add `@testing-library/jest-dom` assertions that **lock current** a11y behavior:
  `toHaveAccessibleName`, `toHaveAccessibleDescription`, `getByRole`, `getByLabelText`,
  `toHaveAttribute('aria-*', ...)`, `toBeDisabled`, `toBeChecked`.
- Tests **must not weaken** existing a11y assertions in any test file.
- Tests **must not** add `aria-*` attributes, `role` attributes, or label text to production
  components in order to make a test easier. If a component is missing an a11y label, file a
  separate defect.
