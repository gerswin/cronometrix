1. **backend/src/auth/middleware.rs** — JWT validation middleware per RESEARCH.md Pattern 4:
   - `pub async fn require_auth(State(state), mut req, next) -> Result<Response, AppError>`:
     a. Extract `Authorization: Bearer <token>` header
     b. Call `verify_access_token(token, state.config.jwt_secret.as_bytes())`
     c. Insert Claims into request extensions: `req.extensions_mut().insert(claims)`
     d. Return Unauthorized if missing/invalid

2. **backend/src/auth/rbac.rs** — Role-gated middleware functions:
   - `pub async fn require_admin(State(state), req, next) -> Result<Response, AppError>`:
     a. Call require_auth logic first
     b. Check claims.role == Role::Admin, return Forbidden if not
   - `pub async fn require_supervisor_or_above(State(state), req, next) -> Result<Response, AppError>`:
     a. Call require_auth logic first
     b. Check claims.role is Admin or Supervisor, return Forbidden if Viewer
   - Helper extractor: `pub struct AuthUser(pub Claims)` — implements `FromRequestParts` by reading Claims from extensions

   Per D-09: Admin = full access, Supervisor = edit subset, Viewer = read-only everything. Map to route groups:
   - Public: /api/v1/setup/*, /api/v1/auth/login, /api/v1/health
   - Authenticated (any role): /api/v1/auth/refresh, /api/v1/auth/logout, GET endpoints
   - Supervisor+: POST/PATCH/DELETE on employees, departments
   - Admin only: POST/PATCH on users, rules

3. **backend/src/auth/handlers.rs** — Auth HTTP handlers:
   - `pub async fn login(State(state), Json(body)) -> Result<impl IntoResponse, AppError>`:
     a. Validate LoginRequest
     b. Query users table: `SELECT id, username, full_name, password_hash, role, status FROM users WHERE username = ?1 AND status = 'active'`
     c. Verify password with `auth::service::verify_password`
     d. Issue access token and refresh token
     e. Hash refresh token, UPDATE users SET refresh_token_hash = ?1 WHERE id = ?2
     f. **REVIEW FIX (LOW): SameSite=Lax instead of Strict** — Build response: JSON body with access_token + user info, Set-Cookie header with refresh token:
        - `httpOnly` = true (XSS protection)
        - `Secure` = true (HTTPS only in production)
        - `SameSite=Lax` (changed from Strict per review — Lax prevents session loss when navigating from third-party links like email or portal in on-premise deployments, while still blocking CSRF POST attacks)
        - `Path=/api/v1/auth` (scoped to auth endpoints only)
        - `Max-Age=604800` (7 days)
     g. Return 200

   - `pub async fn refresh(State(state), jar: CookieJar) -> Result<impl IntoResponse, AppError>`:
     a. Extract refresh token from cookie named "refresh_token"
     b. Verify refresh token JWT
     c. Hash the token, compare to users.refresh_token_hash in DB
     d. Issue NEW access token + NEW refresh token (rotation per D-06)
     e. Update refresh_token_hash in DB
     f. Return new access token in body, new refresh cookie (also SameSite=Lax)

   - `pub async fn logout(State(state), jar: CookieJar) -> Result<impl IntoResponse, AppError>`:
     a. Extract refresh token, verify, get user_id
     b. UPDATE users SET refresh_token_hash = NULL WHERE id = ?1
     c. Clear cookie (Max-Age=0)
     d. Return 200

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /api/v1/auth/login | Untrusted username/password from browser |
| client -> /api/v1/setup/init | Untrusted admin creation payload (first boot only) |
| Cookie -> /api/v1/auth/refresh | Refresh token from httpOnly cookie |
| Bearer header -> middleware | Access token from Authorization header |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | S (Spoofing) | login handler | mitigate | Argon2id hashing via password-auth; timing-safe comparison prevents timing attacks |
| T-01-06 | T (Tampering) | JWT tokens | mitigate | HS256 signature with 32+ char secret; jsonwebtoken validates expiry and signature |
| T-01-07 | R (Repudiation) | auth actions | mitigate | audit_log triggers capture user table mutations; login events logged via tracing |
| T-01-08 | I (Info Disclosure) | error responses | mitigate | Generic "Invalid credentials" on login failure; no username enumeration; raw DB errors not exposed |
| T-01-09 | E (Elevation) | RBAC middleware | mitigate | Role checked from signed JWT claims, not client-supplied; route_layer applied at router group level |
| T-01-10 | T (Tampering) | refresh token | mitigate | SHA-256 hash stored in DB; token rotated on each use; cleared on logout |
| T-01-11 | S (Spoofing) | setup wizard | mitigate | SELECT COUNT(*) + 409 Conflict prevents race condition; endpoint blocked after first admin exists |
| T-01-12 | D (DoS) | password hashing | accept | Argon2id is intentionally slow (~100ms); no rate limiting in Phase 1 (low risk for on-premise) |
| T-01-26 | T (Tampering) | refresh cookie | mitigate | SameSite=Lax blocks CSRF POST attacks while allowing GET navigation from external links; httpOnly prevents XSS token theft; Secure flag enforces HTTPS in production |
</threat_model>
