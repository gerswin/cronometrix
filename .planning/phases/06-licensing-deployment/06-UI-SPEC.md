# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6 frontend work. The only new screen is the license
> activation step (step 0 of the setup wizard). All other Phase 6 work is backend/infrastructure.
> Generated by gsd-ui-researcher.

---

## Scope

Phase 6 introduces **one new frontend screen**: a license activation step that gates the existing
setup wizard. When the system is installed fresh and unlicensed, the user lands on
`/setup/license` before seeing the existing `/setup` admin-account creation screen. After
successful activation the wizard advances to `/setup` as normal.

**Not in scope for this UI-SPEC:**
- The existing `/setup` admin-account page (already built in Phase 1)
- Login page, dashboard, timesheet, reports, devices, employees (already built)
- Backend license middleware (backend-only concern)
- Docker Compose, installer script, DO Functions (no UI)

## Spacing Scale

Declared values (all multiples of 4 — matches Tailwind 4.x default scale):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding (e.g. `gap-1`) |
| sm | 8px | Compact element spacing (e.g. `gap-2`, field gap inside card) |
| md | 16px | Default element spacing (e.g. `gap-4`, card padding) |
| lg | 24px | Section padding (e.g. `gap-6`, CardHeader/CardContent padding) |
| xl | 32px | Layout gaps (not used in this narrow single-card screen) |
| 2xl | 48px | Major section breaks (not used in this phase) |
| 3xl | 64px | Page-level spacing (not used in this phase) |

Exceptions:
- Touch targets for icon-only toggle buttons (show/hide key): minimum 44px hit area via
  `px-3 inset-y-0` pattern already established in `/setup` and `/login` pages.

Precedent: The existing `/setup` and `/login` pages use `flex flex-col gap-4` for form fields
and `p-4` for error banners — both match md (16px). This phase replicates that exact pattern.

## Screen Layout Contract

Route: `/setup/license`

Layout: Full-screen vertically and horizontally centered card — identical to existing `/setup` and
`/login` pages.

```
min-h-screen flex items-center justify-center px-4
  └── Card max-w-md w-full shadow-md
        ├── CardHeader
        │     ├── [Key icon 16px muted-foreground] (decorative, aria-hidden)
        │     ├── CardTitle: "Activate your license"
        │     └── CardDescription: "Enter the license key provided with your Cronometrix installation."
        └── CardContent
              ├── [Error banner — conditional, identical pattern to /setup]
              ├── [Success banner — conditional, shown briefly before redirect]
              └── Form
                    ├── FormField: license_key
                    │     ├── FormLabel: "License key"
                    │     ├── Input (font-mono, uppercase, placeholder: "XXXX-XXXX-XXXX-XXXX")
                    │     └── FormMessage (validation error)
                    └── Button (full-width, type submit)
                          "Activate license" / [Loader2 spin] "Activating…"
```

Flow states:
1. **Checking status** (initial load): Full-screen centered Loader2 spinner — identical to `/setup`
2. **Already activated** (status endpoint returns licensed=true): Redirect to `/setup` or `/login`
   depending on whether admin account exists.
3. **Form idle**: Card with license key input and CTA.
4. **Submitting**: Button shows Loader2 + "Activating…", input disabled.
5. **Error**: Red banner appears above form (same `border-l-4 border-destructive bg-destructive/10` pattern).
6. **Success**: Brief green banner with ShieldCheck "License activated. Continuing setup…" then
   navigate to `/setup` after 1500ms.

## Interaction Contract

### License Key Input

- `type="text"` (not password — operators need to verify they typed it correctly)
- `autoComplete="off"` — license keys are machine-specific, never stored by browser
- `className="font-mono uppercase"` — mono font renders the key characters distinctly
- `maxLength={19}` — enforces XXXX-XXXX-XXXX-XXXX (16 chars + 3 hyphens)
- `spellCheck={false}` — prevents browser underlining of license tokens
- Auto-insert hyphens as user types: on every 4th character input, append `-` automatically
  (optional UX improvement — executor may implement if straightforward, skip if complex)
- `aria-describedby` wired to FormMessage id (same pattern as `/setup`)
- `aria-invalid` set from `fieldState.error` (same pattern as `/setup`)

### Submit Button

- Full-width (`w-full`)
- Disabled state via `aria-disabled` + `onClick` guard (matches `/setup` pattern exactly — do NOT
  use the `disabled` attribute to avoid accessibility flash)
- Shows `Loader2 animate-spin` during submission (16px, `mr-2`)

### Error Banner

- Same pattern as `/setup`: `flex items-center gap-3 p-4 mb-4 rounded border-l-4 border-destructive bg-destructive/10`
- `role="alert"` for screen reader announcement
- `AlertCircle h-4 w-4 text-destructive shrink-0` icon

### Success Banner

- New variant: `flex items-center gap-3 p-4 mb-4 rounded border-l-4 border-green-600 bg-green-50`
  (light mode) / `bg-green-950/30` (dark mode)
- `ShieldCheck h-4 w-4 text-green-600 shrink-0` icon
- Shown for 1500ms, then `router.push("/setup")`

### Status Check on Mount

- `useEffect` on mount: `GET /api/v1/setup/status`
- Response includes both `initialized` (admin account) and `licensed` (license state)
- If `licensed === true` and `initialized === false`: redirect to `/setup` (normal wizard flow)
- If `licensed === true` and `initialized === true`: redirect to `/login`
- If `licensed === false`: show license activation form
- While checking: full-screen `Loader2` spinner (same as `/setup`)
- If backend unreachable: show activation form anyway (allow retry)

## Accessibility Contract

All items below are required, not optional:

| Requirement | Implementation |
|-------------|---------------|
| `role="alert"` on error/success banners | Both banners must have `role="alert"` |
| `aria-invalid` on license key Input | Set from `fieldState.error` via react-hook-form |
| `aria-describedby` on Input → FormMessage | Same `id` linkage as `/setup` |
| `aria-label` on spinner (checking state) | `aria-label="Checking license status"` on Loader2 |
| `aria-disabled` on submitting button | Not `disabled` attribute — matches established pattern |
| Focus management after error | Error banner receives focus or `aria-live="assertive"` to announce |
| Keyboard-only flow | Tab: FormLabel → Input → Button. Enter submits. No mouse required. |
| Color-not-alone rule | Error state uses border + icon + text — not color alone |

## Form Validation Contract (Zod)

Add to `frontend/src/lib/validations.ts`:

```typescript
export const licenseSchema = z.object({
  license_key: z
    .string()
    .min(1, "License key is required.")
    .regex(/^[A-Z0-9]{4}-[A-Z0-9]{4}-[A-Z0-9]{4}-[A-Z0-9]{4}$/i,
      "License key must be in XXXX-XXXX-XXXX-XXXX format."),
})
export type LicenseFormData = z.infer<typeof licenseSchema>
```

Server error codes mapped to banner messages:

| HTTP status | Backend error code | Banner copy |
|------------|-------------------|-------------|
| 404 | `LICENSE_NOT_FOUND` | "License key not found. Check the key and try again." |
| 403 | `HARDWARE_MISMATCH` | "This license is registered to different hardware. Contact support to transfer your license." |
| 409 | `ALREADY_ACTIVATED` | "This license is already active on another installation." |
| 503 / network | any / none | "Could not reach the activation server. Check your internet connection and try again." |
| 422 | `VALIDATION_ERROR` | Display `error.message` from response body |

## API Contract

Endpoint called by the frontend: `POST /api/v1/setup/activate`

Request body:
```json
{ "license_key": "XXXX-XXXX-XXXX-XXXX" }
```

The backend generates the hardware fingerprint server-side — the frontend does NOT send a
fingerprint. The frontend only sends the license key.

Response (success):
```json
{ "activated": true }
```

Response (error): Standard `AppError` shape already used throughout the project.

Status check endpoint (already exists): `GET /api/v1/setup/status`

This endpoint must be extended to include `"licensed": boolean` alongside the existing
`"initialized": boolean` field. The license screen depends on this extension.
