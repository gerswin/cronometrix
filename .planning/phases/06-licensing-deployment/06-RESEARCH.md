## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** RS256 asymmetric signing. Private key lives exclusively on DO Functions. Public key hardcoded as a string constant in Rust source — no env var, no runtime injection. Binary can validate license offline without any secret.
- **D-02:** Public key embedded at compile time in the `license` module. Changing the key requires a recompile.
- **D-03:** Fingerprint composed of 3 components: CPU model+serial (`/proc/cpuinfo`), primary NIC MAC address (`/sys/class/net`), primary disk serial (`/sys/block`). All readable on Linux without root.
- **D-04:** Strict exact match — all 3 components must match the fingerprint recorded at activation time. No N-of-M tolerance.
- **D-05:** Fingerprint = SHA256(cpu_string + mac_string + disk_string). Deterministic, no salt — same hardware always produces the same fingerprint.
- **D-06:** Annual license with soft expiry. JWT `exp` claim = 1 year from activation date.
- **D-07:** Soft expiry behavior: after `exp`, the system keeps running — no API gate re-activates.
- **D-08:** Renewal check at startup when internet is available: if JWT is within 30 days of expiry AND online, call DO Functions to refresh and cache a new JWT. Silent.
- **D-09:** Offline-first: cached JWT is the authority. System operates fully without internet after initial activation.
- **D-10:** Minimal prompts — script asks for exactly 3 things: license key, client slug, admin password.
- **D-11:** Auto-generated by installer: JWT_SECRET (32-byte random hex), DEVICE_CREDS_KEY (32-byte random base64), written to `/opt/cronometrix/.env`.
- **D-12:** Installer sequence: validate license key via DO Functions → generate secrets → write docker-compose.yml + .env → write cloudflared tunnel token to .env → `docker compose up -d`.
- **D-13:** Cloudflare tunnel: pre-created by operator per client. Tunnel token provided to installer. cloudflared service reads `CLOUDFLARE_TUNNEL_TOKEN` from .env. No CF API calls from the installer.

### Claude's Discretion
- Exact fingerprint collection order/parsing details (cpuinfo parsing, which NIC when multiple exist, which disk when multiple exist)
- Docker image tagging and registry choice
- Installer error messages and rollback behavior on failure
- DO Functions implementation language (JS/TS preferred for DO Functions runtime)

### Deferred Ideas (OUT OF SCOPE)
- License dashboard / admin portal for managing multiple client licenses
- Automatic CF tunnel creation via CF Zero Trust API
- License usage analytics (phone-home telemetry)
- Grace period after hardware change (re-activation window)

## Summary

Phase 6 has two independent domains that share only the `.env` file and installer script: (1) the license system (hardware fingerprint + RS256 JWT + offline caching + Tower middleware gate) and (2) the production deployment stack (Docker Compose with Rust API, Next.js frontend, cloudflared). Both domains are well-understood with existing tooling in the project.

The license domain reuses `jsonwebtoken` (already in `Cargo.toml`), adds the `use_pem` feature flag (already in the default feature set — just needs to be explicitly listed alongside `rust_crypto`), and follows the same Tower middleware pattern already used by `require_auth`. Hardware fingerprinting uses `std::fs::read_to_string` on Linux pseudo-filesystems; no new crate is needed since `sha2` is already in the dependency tree. The DO Functions license server is a small Node.js/TypeScript function; `jsonwebtoken` v9 is pre-installed in the Node.js 22 runtime, so no `package.json` is needed.

The deployment domain is straightforward: multi-stage Dockerfiles for both services, a generated `docker-compose.yml` with 3 services, and a bash installer that drives 3 prompts and writes secrets via `openssl rand`. The cloudflared service uses the official `cloudflare/cloudflared` image with the `TUNNEL_TOKEN` environment variable pattern, which is the current recommended approach.

**Primary recommendation:** Build the license module first (fingerprint + JWT cache + middleware gate), then the DO Functions server, then wire the installer to call activation before generating secrets, then write the Dockerfiles.

### Pattern 1: RS256 JWT Offline Validation (Rust)

The public key is embedded at compile time using `include_str!`. A `std::sync::OnceLock` ensures the `DecodingKey` is initialized exactly once per process.

```rust
// Source: jsonwebtoken 10.3.0 docs.rs + verified crates.io feature flags
use jsonwebtoken::{decode, Algorithm, DecodingKey, Validation};
use std::sync::OnceLock;

// Public key embedded at compile time (D-01, D-02)
// Using include_str! so it's a &str — convert to bytes for DecodingKey
const LICENSE_PUBLIC_KEY_PEM: &str = include_str!("pubkey.pem");

static LICENSE_DECODING_KEY: OnceLock<DecodingKey> = OnceLock::new();

pub fn license_decoding_key() -> &'static DecodingKey {
    LICENSE_DECODING_KEY.get_or_init(|| {
        DecodingKey::from_rsa_pem(LICENSE_PUBLIC_KEY_PEM.as_bytes())
            .expect("License public key is invalid PEM — recompile required")
    })
}

pub fn verify_license_jwt(token: &str) -> Result<LicenseClaims, AppError> {
    let mut validation = Validation::new(Algorithm::RS256);
    validation.validate_exp = true; // soft expiry: we still validate but handle it gracefully
    
    let data = decode::<LicenseClaims>(token, license_decoding_key(), &validation)
        .map_err(|_| AppError::Unlicensed)?;
    
    Ok(data.claims)
}
```

[VERIFIED: jsonwebtoken 10.3.0 docs.rs] — `DecodingKey::from_rsa_pem(key: &[u8])` requires `use_pem` feature.
[VERIFIED: crates.io] — `use_pem` and `rust_crypto` can coexist.

### Pattern 3: Hardware Fingerprint Collection

All three sources are world-readable on Linux without root access:

```rust
// Source: Linux kernel docs + verified via manual /proc /sys inspection
use sha2::{Digest, Sha256};
use std::fs;

pub fn collect_fingerprint() -> Result<String, anyhow::Error> {
    // Component 1: CPU model name from /proc/cpuinfo
    // Use first "model name" line; reproducible across reads
    let cpuinfo = fs::read_to_string("/proc/cpuinfo")?;
    let cpu_model = cpuinfo.lines()
        .find(|l| l.starts_with("model name"))
        .and_then(|l| l.split(':').nth(1))
        .map(|s| s.trim().to_string())
        .unwrap_or_default();

    // Component 2: Primary NIC MAC address
    // Primary = first non-loopback interface with a non-zero MAC
    let mac = collect_primary_mac()?;

    // Component 3: Primary block device serial
    // /sys/block/<dev>/device/serial — available without root on most Linux distros
    let disk_serial = collect_primary_disk_serial().unwrap_or_default();

    // D-05: SHA256(cpu_string + mac_string + disk_string) — no salt, deterministic
    let mut hasher = Sha256::new();
    hasher.update(cpu_model.as_bytes());
    hasher.update(mac.as_bytes());
    hasher.update(disk_serial.as_bytes());
    Ok(format!("{:x}", hasher.finalize()))
}

fn collect_primary_mac() -> Result<String, anyhow::Error> {
    // Enumerate /sys/class/net; skip "lo"; take first with valid MAC
    let net_dir = std::fs::read_dir("/sys/class/net")?;
    for entry in net_dir.flatten() {
        let name = entry.file_name().to_string_lossy().to_string();
        if name == "lo" { continue; }
        let mac_path = format!("/sys/class/net/{}/address", name);
        if let Ok(mac) = fs::read_to_string(&mac_path) {
            let mac = mac.trim().to_string();
            if mac != "00:00:00:00:00:00" && !mac.is_empty() {
                return Ok(mac);
            }
        }
    }
    Ok(String::new())
}

fn collect_primary_disk_serial() -> Result<String, anyhow::Error> {
    // /sys/block/<dev>/device/serial — visible without root on most kernels
    let block_dir = std::fs::read_dir("/sys/block")?;
    for entry in block_dir.flatten() {
        let name = entry.file_name().to_string_lossy().to_string();
        // Skip loop devices, ram, etc.
        if name.starts_with("loop") || name.starts_with("ram") { continue; }
        let serial_path = format!("/sys/block/{}/device/serial", name);
        if let Ok(serial) = fs::read_to_string(&serial_path) {
            return Ok(serial.trim().to_string());
        }
    }
    Ok(String::new())
}
```

[ASSUMED] — `/sys/block/<dev>/device/serial` readability without root depends on Linux kernel version and device type. NVMe drives expose serial at `/sys/block/nvme0n1/device/serial`; SATA at `/sys/block/sda/device/serial`. Some virtualized environments (VPS) may return empty or generic serials. The fingerprint will still work — just with an empty component — but uniqueness decreases.

### Pattern 4: DO Functions License Server (Node.js/TypeScript)

```javascript
// Source: DO Functions Node.js 22 runtime; jsonwebtoken v9 pre-installed
// File: do-functions/packages/licenses/activate/index.js

const jwt = require('jsonwebtoken');

// Private key stored as DO Functions environment variable (never in source)
// Set via: doctl serverless deploy with secrets or DO Functions dashboard
const PRIVATE_KEY = process.env.LICENSE_PRIVATE_KEY; // RSA PEM string

exports.main = async function(event) {
    const { license_key, hardware_fingerprint } = event.body || {};
    
    if (!license_key || !hardware_fingerprint) {
        return { statusCode: 400, body: { error: 'Missing required fields' } };
    }
    
    // 1. Look up license record (use environment variable for DB credentials)
    // 2. Validate: key exists + not already bound to different fingerprint
    // 3. Bind fingerprint to license on first activation
    // 4. Sign JWT with RS256
    
    const payload = {
        license_key,
        hardware_fingerprint,
        product: 'cronometrix',
        iat: Math.floor(Date.now() / 1000),
        exp: Math.floor(Date.now() / 1000) + (365 * 24 * 60 * 60), // D-06: 1 year
    };
    
    const token = jwt.sign(payload, PRIVATE_KEY, { algorithm: 'RS256' });
    return { statusCode: 200, body: { token } };
};
```

[VERIFIED: DO docs] — Function endpoint URL format: `https://faas-{region}.doserverless.co/api/v1/web/fn-{namespace-id}/{package}/{function}`.
[VERIFIED: DO docs] — Secrets passed via environment variables; never committed to source. Private key stored as DO Functions env var.

### Pattern 7: Bash Installer (3 prompts)

```bash
#!/usr/bin/env bash
# Source: standard bash installer patterns; openssl rand for secret generation
set -euo pipefail

INSTALL_DIR="/opt/cronometrix"
DATA_DIR="$INSTALL_DIR/data"

echo "=== Cronometrix Installer ==="
read -r -p "License key: " LICENSE_KEY
read -r -p "Client slug (becomes {slug}.cronometrix.com): " CLIENT_SLUG
read -r -s -p "Admin password: " ADMIN_PASSWORD
echo

# Step 1: Collect hardware fingerprint (inline; no separate binary needed)
HARDWARE_FP=$(collect_hardware_fingerprint)  # shell function calling dmidecode or /proc

# Step 2: Validate license against DO Functions
RESPONSE=$(curl -sf -X POST \
  "https://faas-{region}.doserverless.co/api/v1/web/fn-{namespace}/licenses/activate" \
  -H "Content-Type: application/json" \
  -d "{\"license_key\": \"$LICENSE_KEY\", \"hardware_fingerprint\": \"$HARDWARE_FP\"}")

LICENSE_JWT=$(echo "$RESPONSE" | python3 -c "import sys,json; print(json.load(sys.stdin)['token'])")

# Step 3: Generate secrets
JWT_SECRET=$(openssl rand -hex 32)
DEVICE_CREDS_KEY=$(openssl rand -base64 32 | tr -d '\n')

# Step 4: Read Cloudflare tunnel token
read -r -p "Cloudflare tunnel token: " CF_TOKEN

# Step 5: Create install directory and write .env
mkdir -p "$INSTALL_DIR" "$DATA_DIR"
cat > "$INSTALL_DIR/.env" << EOF
JWT_SECRET=$JWT_SECRET
DEVICE_CREDS_KEY=$DEVICE_CREDS_KEY
CLOUDFLARE_TUNNEL_TOKEN=$CF_TOKEN
LICENSE_JWT_PATH=$DATA_DIR/license.jwt
CLIENT_SLUG=$CLIENT_SLUG
TZ=America/Caracas
EOF

# Step 6: Write license JWT to data dir
echo -n "$LICENSE_JWT" > "$DATA_DIR/license.jwt"

# Step 7: Write docker-compose.yml and start
# ... (write template)
docker compose -f "$INSTALL_DIR/docker-compose.yml" up -d

echo "Installation complete. Access at: https://${CLIENT_SLUG}.cronometrix.com"
```

[ASSUMED] — Hardware fingerprint in the installer shell script is auxiliary (for activation call only). The Rust binary re-computes it on startup for anti-cloning check. Shell-based fingerprint collection only needs to match the Rust computation closely enough for activation; the Rust startup check is the authoritative anti-cloning gate.

### Anti-Patterns to Avoid

- **Private key in Rust binary:** DO Functions is the only place the private key exists. The Rust binary only contains the public key. Never commit the private key to the repo.
- **Blocking startup on license check failure:** If the license file doesn't exist, set `license_valid = false` and serve the setup wizard (NOT panic). The system must start to allow first-run activation.
- **Validation only at activate time:** Re-validate fingerprint at every startup (not just activation). Anti-cloning protection requires the running machine to match — not just the activating machine.
- **Soft expiry mishandling:** After JWT `exp`, the gate stays OPEN (D-07). Do not re-activate the gate on expiry. Only the renewal path changes — the production system keeps running.
- **Storing cloudflared credentials file:** Use `TUNNEL_TOKEN` env var pattern (not `--credentials-file`). Operator provides token from CF Zero Trust dashboard; no CF API integration needed.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| RS256 JWT signing/verification | Custom RSA + JWT impl | `jsonwebtoken` crate (already in project) | Edge cases in ASN.1 encoding, algorithm confusion attacks |
| Secret generation | `rand::thread_rng` + manual hex | `openssl rand -hex 32` in installer | `openssl rand` is simpler in bash; for Rust startup, `rand` is fine |
| Hardware fingerprint crate | Custom `/proc` parser from scratch | `std::fs` reads — no crate needed | `sha2` already in Cargo.toml; file paths are stable Linux interfaces |
| Docker image registry | Self-hosted registry | GitHub Container Registry (ghcr.io) or Docker Hub | No infra to maintain; ghcr.io is free for public images |
| Cloudflare tunnel management | Custom HTTP proxy + TLS cert management | `cloudflare/cloudflared` official image | Handles cert renewal, reconnects, Zero Trust routing automatically |
| DO Functions persistence | Custom server or database | DO Functions env vars + any DB as needed | For v1: license records can live in a DO managed Postgres (cheapest: $15/mo) or simple JSON in DO Spaces |

**Key insight:** The license system's security comes from the RS256 asymmetric key pair, not from novel crypto implementation. Use existing well-audited libraries; the custom work is only the business logic (fingerprint matching, activation flow).

## Common Pitfalls

### Pitfall 1: `use_pem` feature not listed alongside `rust_crypto`
**What goes wrong:** `DecodingKey::from_rsa_pem()` does not compile — function doesn't exist.
**Why it happens:** `use_pem` is NOT included in `rust_crypto` feature. They are independent. Current Cargo.toml has only `features = ["rust_crypto"]`.
**How to avoid:** Change to `features = ["rust_crypto", "use_pem"]` in Cargo.toml.
**Warning signs:** `error[E0599]: no function or associated item named 'from_rsa_pem' found for struct 'DecodingKey'`.

### Pitfall 2: Validation `validate_exp = true` panics after soft expiry
**What goes wrong:** System re-gates itself after license `exp` date, contradicting D-07.
**Why it happens:** Default `Validation` in jsonwebtoken has `validate_exp = true`, which returns `Err` on expired tokens.
**How to avoid:** On startup license check, set `validation.validate_exp = false` (soft expiry means expiry doesn't block). Still verify the signature. Add a separate expiry-within-30-days check for renewal.
**Warning signs:** After 1 year, system suddenly stops serving requests.

### Pitfall 3: Hardware fingerprint differs between installer and Rust startup check
**What goes wrong:** License activates successfully but system rejects its own license on startup.
**Why it happens:** Bash fingerprint collection (in installer) uses slightly different parsing than Rust fingerprint collection — e.g., bash trims differently, or selects a different NIC.
**How to avoid:** Option A: Installer calls the Rust binary's `fingerprint` subcommand to generate the fingerprint (requires binary built before install). Option B: Keep fingerprint collection only in Rust; installer calls a thin Rust binary or the already-built Docker image. Option C: Make installer fingerprint "close enough" — the anti-cloning gate is in Rust; the installer fingerprint is only used to pre-bind the license.
**Warning signs:** Activation returns 200, but startup logs show `fingerprint_mismatch`.

### Pitfall 4: `/sys/block` serial empty on VPS/cloud instances
**What goes wrong:** Disk serial returns empty string on VirtualBox/KVM/cloud VMs; fingerprint degrades to SHA256(cpu + mac + "").
**Why it happens:** Virtual disks often don't expose serial numbers via sysfs. NVMe vs SATA vs virtio have different serial paths.
**How to avoid:** D-05 allows this — empty component still produces a valid (though less unique) fingerprint. Document this as expected behavior for VPS deployments. Fingerprint will still be unique if CPU model + MAC are unique.
**Warning signs:** Not a failure — just lower uniqueness. Flag in docs.

### Pitfall 5: Docker image build context includes secrets
**What goes wrong:** `.env` file or `pubkey.pem` gets baked into Docker image layers.
**Why it happens:** `COPY . .` in Dockerfile copies everything including `.env`.
**How to avoid:** Add `.env`, `*.pem`, `data/` to `.dockerignore`. Public key PEM is embedded via `include_str!` at compile time — the `pubkey.pem` file is only needed during `cargo build`, not at runtime image.
**Warning signs:** `docker history` shows secrets in image layers.

### Pitfall 6: `docker compose up` fails silently if `.env` is missing
**What goes wrong:** `CLOUDFLARE_TUNNEL_TOKEN` is unset; cloudflared starts but connects to wrong/no tunnel.
**Why it happens:** Docker Compose silently substitutes empty strings for missing env vars by default.
**How to avoid:** Use `${CLOUDFLARE_TUNNEL_TOKEN:?CLOUDFLARE_TUNNEL_TOKEN is required}` syntax in docker-compose.yml or validate `.env` completeness in the installer before calling `docker compose up`.
**Warning signs:** cloudflared container exits with authentication error.

### Pitfall 7: Installer not idempotent — secrets regenerated on re-run
**What goes wrong:** Re-running installer generates new JWT_SECRET, invalidating all existing user sessions.
**Why it happens:** `openssl rand` generates new values each run.
**How to avoid:** Installer checks for existence of `$INSTALL_DIR/.env` and skips secret generation if present. Only re-run the activation call if license.jwt is missing.
**Warning signs:** Users are logged out after re-install.

```rust
pub async fn load_and_validate_license(config: &Config) -> bool {
    let jwt_path = &config.license_jwt_path;
    let token = match std::fs::read_to_string(jwt_path) {
        Ok(t) => t.trim().to_string(),
        Err(_) => return false,  // no license file — first run
    };
    
    let claims = match verify_license_jwt(&token) {
        Ok(c) => c,
        Err(_) => return false,  // invalid signature
    };
    
    // Anti-cloning: re-compute fingerprint and compare
    let current_fp = match collect_fingerprint() {
        Ok(fp) => fp,
        Err(_) => return false,
    };
    
    if claims.hardware_fingerprint != current_fp {
        tracing::error!("License fingerprint mismatch — hardware may have changed");
        return false;
    }
    
    true
}
```

## Open Questions (RESOLVED)

1. **Installer fingerprint vs. Rust fingerprint alignment**
   - What we know: Both read the same Linux files; parsing differences can cause mismatch.
   - What's unclear: Whether to ship a pre-built fingerprint binary in the installer or implement collection twice.
   - **RESOLVED:** live-API activation. The installer brings up `docker compose up -d api` first, then drives `POST /api/v1/setup/activate` against the running API container with the license key. The Rust binary is the single fingerprint authority — the installer never computes a fingerprint itself. This eliminates Pitfall 3 (installer/Rust fingerprint mismatch) entirely. See Plan 03 install.sh for the implementation.

2. **DO Functions license record storage**
   - What we know: DO Functions supports env vars for secrets; needs a DB for license records.
   - What's unclear: Which DB the operator uses for the license server (DO Managed Postgres, PlanetScale, simple JSON in DO Spaces, etc.).
   - **RESOLVED:** Postgres-backed store (DO Managed Postgres). Single `licenses` table keyed on `license_key TEXT PRIMARY KEY` with `hardware_fingerprint`, `activated_at`, `last_renewed_at` columns. DO Functions handlers use the `pg` npm package; connection string supplied via `DATABASE_URL` env var. See Plan 04 README and shared-store.js test harness.

3. **Docker image registry for distribution**
   - What we know: Need to distribute `cronometrix/api` and `cronometrix/web` images to client servers.
   - What's unclear: Public Docker Hub vs. private GitHub Container Registry (ghcr.io).

## Security Domain

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | yes | License gate + JWT verification before any API access |
| V3 Session Management | no | License is machine-scoped, not session-scoped |
| V4 Access Control | yes | `require_license` middleware before `require_auth`; public key hardcoded (not injectable) |
| V5 Input Validation | yes | License key input: length/format validation before calling DO Functions |
| V6 Cryptography | yes | RS256 asymmetric signing; no symmetric secret in binary; `sha2` for fingerprint |

### Known Threat Patterns for License System

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| License JWT replay on different hardware | Spoofing | Hardware fingerprint claim in JWT; Rust startup re-computes and compares |
| Private key extraction from binary | Elevation of Privilege | Private key is NEVER in the binary (D-01); only public key embedded |
| Algorithm confusion attack (alg=HS256 swap) | Tampering | `Validation::new(Algorithm::RS256)` hardcodes algorithm; jsonwebtoken rejects alg header mismatches |
| License file tampering | Tampering | RS256 signature covers all claims including hardware_fingerprint; tampered JWT fails signature check |
| DO Functions endpoint spoofing | Spoofing | HTTPS + CA-verified certificate chain; use `rustls-tls` feature (already in reqwest config) |
| Installer secrets in shell history | Information Disclosure | Use `read -s` for passwords; avoid echoing secrets to stdout in installer |
| Docker image layer containing secrets | Information Disclosure | `.dockerignore` excludes `.env` and `data/`; public key only (not secret) in source tree |
