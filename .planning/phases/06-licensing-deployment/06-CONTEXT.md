# Phase 6: Licensing & Deployment - Context

**Gathered:** 2026-04-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Hardware-bound license gate that blocks API access until a valid license is activated, plus a one-command `curl | bash` installer that sets up Docker Compose (api + web + cloudflared) and connects the installation to a pre-created Cloudflare tunnel. Each client installation is fully independent.

</domain>

<decisions>
## Implementation Decisions

### License JWT Signing
- **D-01:** RS256 asymmetric signing. Private key lives exclusively on DO Functions (license server). Public key hardcoded as a string constant in Rust source — no env var, no runtime injection. Binary can validate license offline without any secret.
- **D-02:** Public key embedded at compile time in the `license` module. Changing the key requires a recompile (acceptable — key rotation is rare for hardware-bound licenses).

### Hardware Fingerprint
- **D-03:** Fingerprint composed of 3 components: CPU model+serial (`/proc/cpuinfo`), primary NIC MAC address (`/sys/class/net`), primary disk serial (`/sys/block`). All readable on Linux without root.
- **D-04:** Strict exact match — all 3 components must match the fingerprint recorded at activation time. If any hardware changes, license is rejected and client must contact support to re-activate. No N-of-M tolerance.
- **D-05:** Fingerprint = SHA256(cpu_string + mac_string + disk_string). Deterministic, no salt — same hardware always produces the same fingerprint.

### License Renewal Model
- **D-06:** Annual license with soft expiry. JWT `exp` claim = 1 year from activation date.
- **D-07:** Soft expiry behavior: after `exp`, the system **keeps running** — no API gate re-activates. Expiry ends the support+updates entitlement only, not production access.
- **D-08:** Renewal check at startup when internet is available: if JWT is within 30 days of expiry AND online, call DO Functions to refresh and cache a new JWT. Silent — no user-facing prompt.
- **D-09:** Offline-first: cached JWT is the authority. System operates fully without internet after initial activation (DEPL-04).

### Installer Script
- **D-10:** Minimal prompts — script asks for exactly 4 things: license key, client slug (becomes `{slug}.cronometrix.com` subdomain), admin password, and Cloudflare tunnel token. Note: D-13 requires the tunnel token at install time (cloudflared cannot start without it), hence the 4th prompt. Nothing else interactive.
- **D-11:** Auto-generated by installer: JWT_SECRET (32-byte random hex), DEVICE_CREDS_KEY (32-byte random base64), all written to `/opt/cronometrix/.env`.
- **D-12:** Installer sequence: generate secrets → write `.env` (including `CLOUDFLARE_TUNNEL_TOKEN`, `DO_FUNCTIONS_ACTIVATE_URL`, `DO_FUNCTIONS_RENEW_URL`) → write `docker-compose.yml` → `docker compose up -d api` → `POST /api/v1/setup/activate` against the running API container (the API computes the canonical fingerprint and calls DO Functions itself; this provides a single fingerprint authority and eliminates RESEARCH Pitfall 3 — installer/Rust fingerprint mismatch) → `docker compose up -d web cloudflared`. Rationale: live-API activation makes the Rust binary the sole source of truth for the fingerprint; the installer never re-implements `/proc`/`/sys` parsing.
- **D-13:** Cloudflare tunnel: pre-created by operator in CF Zero Trust dashboard per client. Tunnel token provided to installer prompt (or passed as arg). cloudflared service reads `CLOUDFLARE_TUNNEL_TOKEN` from .env. No CF API calls from the installer.

### Claude's Discretion
- Exact fingerprint collection order/parsing details (cpuinfo parsing, which NIC when multiple exist, which disk when multiple exist)
- Docker image tagging and registry choice
- Installer error messages and rollback behavior on failure
- DO Functions implementation language (JS/TS preferred for DO Functions runtime)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `backend/src/auth/middleware.rs` (`require_auth`): License gate middleware will follow the same Tower layer pattern — validate license JWT from `AppState`, reject with 403 if unlicensed
- `backend/src/config.rs` (`Config::from_env()`): Add `license_jwt_path` (path to cached license file) to Config; same env var loading pattern
- `backend/src/errors.rs` (`AppError`): Add `Unlicensed` variant mapping to HTTP 403

### Established Patterns
- Secrets never printed in Debug impls (see Config debug redaction) — license private key on DO Functions must never appear in logs
- Tower middleware layers applied per router group — license gate is a new layer wrapping all non-setup routes
- JWT library: `jsonwebtoken` crate already in Cargo.toml — add RS256 algorithm support (DecodingKey::from_rsa_pem)

### Integration Points
- `backend/src/main.rs`: License gate middleware added to the router builder, applied before `require_auth`
- `backend/src/setup/`: License activation endpoint (`POST /setup/activate`) slots into the existing setup wizard flow
- Docker Compose: new `cloudflared` service added alongside existing `api` and `web` services

</code_context>

<specifics>
## Specific Ideas

- License activation UI is part of the existing setup wizard (Phase 1 foundation). License key entry = step 0 before any other config.
- DO Functions license server validates: fingerprint matches stored record for that license key + license not already activated on another machine.
- Installer installs to `/opt/cronometrix/` by convention.
- cloudflared service: use official `cloudflare/cloudflared` Docker image, not a custom build.

</specifics>

<deferred>
## Deferred Ideas

- License dashboard / admin portal for managing multiple client licenses — out of scope (each install is independent)
- Automatic CF tunnel creation via CF Zero Trust API — deferred; pre-created tunnel token is sufficient for v1
- License usage analytics (phone-home telemetry) — out of scope for v1
- Grace period after hardware change (re-activation window) — defer to support process, not a system feature

</deferred>
