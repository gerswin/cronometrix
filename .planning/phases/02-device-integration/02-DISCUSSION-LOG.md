## Credential Storage

### Q: How should ISAPI device passwords be stored in the database?

| Option | Description | Selected |
|--------|-------------|----------|
| AES-GCM with app key (Recommended) | Encrypt with AES-256-GCM; key from env var. Decrypt in memory only when dispatching. | ✓ |
| Plaintext column | Store raw passwords. Trusts on-prem server fully; trivial to leak if DB is copied. | |
| OS keyring per device | Linux Secret Service / macOS Keychain. Breaks in Docker / headless servers. | |

### Q: Where does the encryption key come from?

| Option | Description | Selected |
|--------|-------------|----------|
| Separate DEVICE_CREDS_KEY env var (Recommended) | Independent 32-byte key, generated by installer. Isolated from JWT_SECRET. | ✓ |
| Derived from JWT_SECRET via HKDF | Reuse JWT_SECRET; one fewer env var but couples security domains. | |
| Key file on disk (mode 600) | Separate file, mode 600. Operator backs up independently. | |

### Q: What does the API return for a device's password field?

| Option | Description | Selected |
|--------|-------------|----------|
| Never return password (Recommended) | Field absent from all responses. PATCH to change; no view flow. | ✓ |
| Return masked (****) | Fixed mask for "password configured" indication. | |
| Return decrypted to Admin only | Separate admin-only reveal endpoint. Widens admin-token blast radius. | |

### Q: Key rotation policy for Phase 2?

| Option | Description | Selected |
|--------|-------------|----------|
| Defer rotation to later phase (Recommended) | Phase 2 = encrypt/decrypt only. Rotation tooling later. | ✓ |
| Include re-encrypt CLI now | Ship rotation CLI in Phase 2. | |

## Dedup & Unknown Faces

### Q: The 30-second dedup window filters duplicates by what key?

| Option | Description | Selected |
|--------|-------------|----------|
| (employee_id, device_id, direction) (Recommended) | Dedup per device+direction. Multi-device events both persist; Phase 3 picks canonical. | ✓ |
| (employee_id, direction) | Dedup across all devices. Loses multi-device audit signal. | |
| (employee_id, device_id) ignore direction | Merges quick entry→exit toggles. | |

### Q: Hikvision sends an event whose face_id has no matching employee. What happens?

| Option | Description | Selected |
|--------|-------------|----------|
| Store with employee_id=NULL + flag unknown (Recommended) | Persist with is_unknown=1 + captured face_id. Reconciliation possible later. | ✓ |
| Silently discard with warn log | Drop event. Loses "unknown person" dashboard signal. | |
| Reject via ISAPI (block door) | Door access is device-firmware decision; Cronometrix does not participate. | |

### Q: Where does dedup logic live?

| Option | Description | Selected |
|--------|-------------|----------|
| UNIQUE index + INSERT OR IGNORE (Recommended) | Composite UNIQUE (employee_id, device_id, direction, 30s-bucket). Atomic; no races. | ✓ |
| Pre-insert SELECT check in Rust | Open to races across tokio tasks; needs mutex. | |
| In-memory LRU per device | No DB hit but dies on restart; no cross-device dedup. | |

### Q: face_id → employee mapping storage:

| Option | Description | Selected |
|--------|-------------|----------|
| employees.external_face_id column (Recommended) | Single column; assumes one face_id per employee across all devices. | |
| Separate device_face_mappings table | (device_id, face_id) → employee_id. Handles per-device face_id variance. | ✓ |

**Notes:** User overrode the recommendation — chose the more flexible per-device mapping table to accommodate Hikvision's potentially device-specific face_id assignment.

## ISAPI Command Dispatch

### Q: ISAPI command dispatch — sync or async?

| Option | Description | Selected |
|--------|-------------|----------|
| Synchronous with timeout (Recommended) | Handler blocks until device replies or 10s timeout. 200 or 504. | ✓ |
| Async queue + status endpoint | 202 + command_id; poll status. More code; operator confusion. | |
| Hybrid sync for quick, async for enrollment | Mixed latency paths; more code. | |

### Q: HTTP request timeout for sync ISAPI commands?

| Option | Description | Selected |
|--------|-------------|----------|
| 10 seconds (Recommended) | Headroom over typical ISAPI latency; 504 on exceed. | ✓ |
| 5 seconds | Tight; may false-timeout. | |
| 30 seconds | Generous; ties up HTTP worker. | |

### Q: Are ISAPI commands audited?

| Option | Description | Selected |
|--------|-------------|----------|
| Yes — dedicated command_audit_log table (Recommended) | Who/device/command/result/timestamp. Forensic coverage for door-open events. | ✓ |
| Yes — reuse existing audit_log via triggers | Commands aren't table mutations; triggers don't fire naturally. | |
| No | Fails audit-everything principle. | |

### Q: API shape for dispatching commands?

| Option | Description | Selected |
|--------|-------------|----------|
| POST /devices/:id/commands { command: "door_open" } (Recommended) | Single endpoint; extensible enum. | ✓ |
| POST /devices/:id/door/open + /reboot + /enrollment-mode | Dedicated routes per action. More verbose routing. | |

## Event Payload Retention

### Q: Keep the raw alertStream XML payload?

| Option | Description | Selected |
|--------|-------------|----------|
| Yes, store in attendance_events.raw_xml (Recommended) | ~2–5KB/event. Priceless for schema-drift debugging (the flagged blocker). | ✓ |
| No — parse only, discard XML | Smallest DB; loses debugging signal on new device models. | |
| Store with 30-day TTL | Nightly purge; middle ground; adds cron burden. | |

### Q: How to store the JPEG face capture?

| Option | Description | Selected |
|--------|-------------|----------|
| Filesystem + path in DB (Recommended) | ./data/events/YYYY-MM-DD/{event_id}.jpg. Keeps SQLite and Turso sync small. | ✓ |
| BLOB in SQLite | Everything in one file but bloats Turso sync (~40KB/event). | |
| Discard JPEGs entirely | Kills photo-evidence audit trail for timesheet disputes. | |

### Q: Retention policy for attendance_events?

| Option | Description | Selected |
|--------|-------------|----------|
| Keep forever + document later (Recommended) | Indefinite retention; events are legal records. | ✓ |
| 90-day raw_xml purge, keep parsed fields | Trims storage; keeps legal record. | |
| Configurable retention via global_rules | Needs UI; feature creep for Phase 2. | |

### Q: Is there a Phase 2 API to READ attendance events?

| Option | Description | Selected |
|--------|-------------|----------|
| Minimal GET /events with pagination + filters (Recommended) | Phase 3 and Phase 4 consumers can query; same RBAC/conventions as Phase 1. | ✓ |
