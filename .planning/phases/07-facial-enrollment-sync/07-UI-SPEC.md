# Phase 7 — UI Design Contract

> Visual and interaction contract for the Enrolamiento Facial screen and modal.
> Mockup `d7atd` in `untitled.pen` is the locked visual reference. Generated by
> gsd-ui-researcher, verified by gsd-ui-checker.

## Scope

Phase 7 produces **one screen + one modal**:

1. `/enrollment` route page — replaces the existing placeholder body (Phase 4 D-12). Lists in-progress enrollments and exposes the modal trigger via an employee picker.
2. `EnrollmentModal` — the centerpiece. Three-tab capture (Lector Hikvision / Webcam / Subir JPG), AI Validation panel right-side, per-device Sincronización a Dispositivos panel below validation, "Capturar Rostro" / "Aceptar" CTA at bottom.
3. Per-row "Enrolar Rostro" action wired into the existing Employees table (`employees/page.tsx`). No new screen, only an action button — same modal opens with that employee preselected.

**Not in scope for this UI-SPEC:**
- Mass admin re-sync screen (deferred — see CONTEXT.md Deferred Ideas)
- Enrollment audit panel UI (audit data persists via triggers; viewer is a future phase)
- Mobile / responsive layouts (Phase 4 D-3 desktop-only ≥1280px carries forward)
- Backend ISAPI implementation (07-01 plan, no UI surface)

## Color

Pre-populated from `frontend/src/app/globals.css` CSS custom properties (OKLCH, neutral base). All values flow through CSS variables — Phase 7 must never hard-code OKLCH or hex.

| Role | CSS Variable | Light value | Dark value | Usage in this phase |
|------|-------------|-------------|------------|---------------------|
| Dominant (60%) | `--background` | `oklch(1 0 0)` | `oklch(0.145 0 0)` | Page background, modal overlay base |
| Secondary (30%) | `--card` / `--muted` | `oklch(1 0 0)` / `oklch(0.97 0 0)` | `oklch(0.205 0 0)` / `oklch(0.269 0 0)` | DialogContent surface, validation panel surface, sync panel surface, tab inactive backgrounds |
| Accent (10%) | `--primary` | `oklch(0.205 0 0)` | `oklch(0.922 0 0)` | Primary CTA button only ("Capturar Rostro", "Aceptar", "Iniciar Captura"), active tab indicator |
| Destructive | `--destructive` | `oklch(0.577 0.245 27.325)` | `oklch(0.704 0.191 22.216)` | Failed sync row icon + bar, validation-failed icon, kiosk timeout banner, AppError toast border-left |

Accent reserved for (explicit list — no other elements use accent):
- DialogFooter primary CTA: "Capturar Rostro" (webcam) / "Aceptar" (preview after capture) / "Iniciar Captura" (kiosk start) / "Reintentar" (kiosk timeout retry)
- Active tab visual indicator (border-bottom or background of active TabsTrigger)
- Focused input ring (browser/Base UI default `--ring`)

State color tokens (sync rows + AI Validation panel) — these are NOT new accent colors, they are semantic state indicators reusing the carry-forward palette already used by `device-table.tsx` `StatusBadge`:

| State | Background | Foreground / Icon | Tailwind classes |
|-------|------------|-------------------|------------------|
| Pending / Evaluating | `oklch(0.97 0 0)` (muted) | `oklch(0.556 0 0)` (muted-foreground, slate-500 equivalent) | `bg-slate-100 text-slate-600` (matches `StatusBadge` `unknown`) |
| Success / Pass | green-100 / green-700 | `text-green-700` + `CheckCircle2` icon | `bg-green-100 text-green-700` (matches `StatusBadge` `online`) |
| Failure / Fail | red-100 / red-700 | `text-red-700` + `XCircle` icon | `bg-red-100 text-red-700` (matches `StatusBadge` `offline`) |
| Warning (kiosk timeout) | amber-50 / amber-600 | `text-amber-600` | `bg-amber-50 text-amber-600` (matches `command-modal.tsx` reboot warning) |

Rationale: The 60/30/10 split is preserved at the layout level (background dominant, cards/panels secondary, primary button accent). State colors are functional indicators on small surfaces (badges, progress bars, validation chips) — they do not compete with accent for attention because they are distributed at low frequency across discrete state markers.

## Interaction Contract

### State machine (modal lifecycle)

```
idle  →  capturing  →  preview  →  submitting  →  syncing  →  terminal{success|partial|failed}
                ↓ (recapturar)
                ←
```

- **idle:** modal just opened. Tabs default to "Lector Hikvision". Validation panel shows "Inicia la captura para evaluar.". Sync panel lists devices in "Esperando…". Footer CTA: "Iniciar Captura" or "Capturar Rostro" depending on tab.
- **capturing:** kiosk waiting OR webcam streaming OR upload in-progress. Validation panel evaluates as frames arrive (webcam) or once upload completes (upload). Footer CTA disabled until all three validations green.
- **preview:** captured frame frozen, all three validations green. Footer CTA: "Aceptar" (primary).
- **submitting:** POST `/api/v1/enrollments` in flight. Footer CTA shows Loader2 + "Enviando…". Cerrar enabled (D-09).
- **syncing:** got 202 + enrollment_id, polling `GET /api/v1/enrollments/:id` every 1500ms. Sync rows update to "Enviando…" then terminal. Footer helper: "Enviando a {N} dispositivos…".
- **terminal:** all sync rows in success/failed. Footer helper renders the right summary line. Cerrar remains enabled. Per-row "Reintentar" appears on failed rows.

### Polling behavior (D-07 carry-forward)

- `useQuery({ queryKey: ['enrollment', id], refetchInterval: 1500 })`.
- Stop polling when every push row is terminal — implement via `refetchInterval: (data) => allTerminal(data) ? false : 1500`.
- On modal close mid-sync: query keeps polling (D-09). When background query resolves to terminal, fire one of the three Sonner toasts above.

### Validation gate (D-05 hard gate)

- "Aceptar" / "Capturar Rostro" footer button is disabled (`aria-disabled="true"`, NOT HTML `disabled` per Phase 1 D-09 carry-forward) when ANY of the three validation rows is not "OK".
- Disabled state visual: opacity 50, cursor not-allowed, pointer-events none — handled by shadcn Button's existing `aria-disabled` styling.
- Tooltip is NOT required (face-api.js evaluates ~10× per second on webcam; the validation panel itself communicates the blocker in real time).

### Per-row retry (D-08)

- Each `SyncRow` in `failed` state renders an inline "Reintentar" Button (variant outline, size sm).
- Click → `POST /api/v1/enrollments/:id/devices/:device_id/retry`.
- On 202: row state flips to `pending` → `in_progress` (next poll) → terminal. No optimistic UI; rely on poll.
- Multiple concurrent retries allowed (each row independent).

### Modal close (D-09)

- "Cerrar" Button + `Esc` key + clicking overlay all call `onOpenChange(false)`.
- Modal close NEVER aborts the submit mutation or the polling query.
- If state was `submitting` or `syncing` at close time, fire a sticky Sonner toast: "Enrolamiento en curso — {success}/{total} dispositivos" with id `enrollment-${id}` so subsequent terminal toast can dismiss + replace it.
- Re-opening the modal (via "Ver detalles" on `/enrollment` row) re-attaches to the same query — no duplicate work.

### face-api.js lazy load (Specifics)

- Skeleton + "Cargando modelo de IA…" copy ONLY on the very first time the modal opens in a session (subsequent opens use the cached model).
- Implement via dynamic `import('face-api.js')` inside `useEffect` of the modal — never at route or app level. CONTEXT.md mandates this so admins who never open the modal don't pay the 6 MB cost.

### Background workers visibility (D-15 / D-16)

These backend workers (purge on employee deactivate, backfill on new device registration) DO NOT have a Phase 7 UI surface. The "Sincronización Inicial — X/Y empleados" panel mentioned in CONTEXT.md D-16 lives on the **device detail page** which is part of the existing Phase 4 device manager — Phase 7 only adds a backend hook. ## API Contract (UI consumption surface)

The UI calls four backend endpoints. Backend shape comes from CONTEXT.md `decisions` and 07-RESEARCH.md (when ready); this section documents only the UI's expectations.

| Endpoint | Method | UI behavior |
|----------|--------|-------------|
| `/api/v1/enrollments/capture-from-device` | POST | Body `{ device_id, employee_id }`. Returns 202 + `{ capture_id }`; subsequent poll on `GET /api/v1/enrollments/captures/:capture_id` until status terminal — UI shows kiosk waiting state. On terminal `captured`: response includes `{ photo_url, photo_blob_path }` for preview. (If planner decides to inline the JPG in the response payload as base64, UI accepts both shapes.) |
| `/api/v1/enrollments` | POST (multipart) | UI submits `FormData` with `employee_id`, `captured_via`, optional `source_device_id`, `photo` (Blob). On 202: `{ enrollment_id, device_pushes: [...] }` — UI begins polling. |
| `/api/v1/enrollments/:id` | GET | Polled every 1500ms. Response: `{ id, employee_id, status, started_at, completed_at, device_pushes: [{ device_id, device_name, status, error_message, started_at, completed_at }] }`. |
| `/api/v1/enrollments/:id/devices/:device_id/retry` | POST | UI invokes on per-row "Reintentar" click. 202 → next poll surfaces in-progress. |

Error envelope reuses Phase 1 standard `AppError` shape: `{ code, message, details? }`. UI maps `code` to inline copy when available; falls back to `message` then to generic toast text.
